## Логика SLA

### Параметры SLA по типу договора

| Тип договора | Время реакции | Время решения |
|---|---|---|
| Премиум | ≤ 2 часа | ≤ 24 часа |
| Стандарт | ≤ 8 часов | ≤ 72 часа |
| Нет договора | ≤ 8 часов | ≤ 72 часа (по умолчанию) |

### Дедлайны

| Метрика | Отсчёт от | Отсчёт до |
|---|---|---|
| Реакция | `created_at` | `assigned_at` |
| Решение | `created_at` | перевод в статус «Выполнена» |

---

## Основной поток — Мониторинг и эскалация

1. Фоновый процесс каждую минуту проверяет все заявки со статусом не в {Выполнена, На проверке, Закрыта}.
2. **За 1 час до дедлайна реакции** (заявка ещё не назначена):
   - Система отправляет Руководителю сервиса email + in-app уведомление:
     *«Заявка #[N] — до нарушения SLA реакции 1 час. Назначьте инженера»*.
3. **За 4 часа до дедлайна решения** (заявка не переведена в «Выполнена»):
   - Система отправляет Руководителю сервиса email + in-app уведомление:
     *«Заявка #[N] — до нарушения SLA решения 4 часа»*.
4. **При нарушении SLA реакции** (заявка не назначена после дедлайна):
   - Система устанавливает `sla_reaction_violated = true`.
   - В карточке заявки отображается красный индикатор **«SLA реакции нарушен»**.
5. **При нарушении SLA решения** (заявка не выполнена после дедлайна):
   - Система устанавливает `sla_resolution_violated = true`.
   - В карточке заявки отображается красный индикатор **«SLA решения нарушен»**.

## Просмотр SLA в карточке заявки

Пользователь видит в карточке заявки блок SLA:
- **Дедлайн реакции:** [дата/время] — [статус: зелёный / жёлтый / красный]
- **Дедлайн решения:** [дата/время] — [статус: зелёный / жёлтый / красный]
- **Оставшееся время:** countdown до ближайшего дедлайна

Цветовая индикация:
- 🟢 Более 4 часов до дедлайна
- 🟡 Менее 4 часов до дедлайна
- 🔴 Дедлайн нарушен

## Acceptance Criteria

```gherkin
Feature: SLA-контроль и эскалация

  Scenario: Эскалация за 1 час до нарушения SLA реакции
    Given заявка создана, не назначена, sla_reaction_deadline = через 55 минут
    When фоновый процесс выполняет проверку
    Then Руководитель сервиса получает email и in-app уведомление об эскалации
    And уведомление содержит номер заявки и «до нарушения SLA реакции 1 час»

  Scenario: Фиксация нарушения SLA реакции
    Given заявка не назначена, sla_reaction_deadline наступил 10 минут назад
    When фоновый процесс выполняет проверку
    Then sla_reaction_violated = true
    And в карточке заявки отображается индикатор «SLA реакции нарушен»

  Scenario: Эскалация за 4 часа до нарушения SLA решения
    Given заявка в статусе «В работе», sla_resolution_deadline = через 3 часа 50 минут
    When фоновый процесс выполняет проверку
    Then Руководитель сервиса получает уведомление «до нарушения SLA решения 4 часа»

  Scenario: SLA не нарушается для заявки в финальном статусе
    Given заявка в статусе «Закрыта»
    When фоновый процесс выполняет проверку
    Then для этой заявки эскалация не отправляется и флаги не меняются
```

## Примечания / Бизнес-правила

- **BR-F-903:** значения SLA берутся из договора клиента (Модуль 13). При отсутствии договора применяются значения Стандарт.
- **BR-F-904:** эскалация — только Руководителю сервиса. Инженер не уведомляется напрямую об эскалации.
- Уведомления эскалации отправляются однократно (не повторяются каждую минуту).
- Флаги нарушения `sla_reaction_violated` и `sla_resolution_violated` не сбрасываются даже после назначения/выполнения — они фиксируют факт нарушения для отчётности (→ BR-F-908).
