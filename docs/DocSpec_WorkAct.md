## 2. Состояния

| Код | Отображаемое название | Описание |
| --- | --- | --- |
| `draft` | Черновик | Акт создан, редактирование разрешено |
| `signed` | Подписан | Акт подписан клиентом, редактирование запрещено |

---

## 3. Переходы между состояниями

| Из → В | Условие | Кто выполняет | Проверки |
| --- | --- | --- | --- |
| `draft` → `signed` | Клиент подписывает акт | client_user | Пользователь принадлежит организации заявки; заявка не отменена |

> Обратного перехода `signed` → `draft` не существует. Подписанный акт редактировать нельзя.

## 7. Проверки при переходах

| Переход | Проверки |
| --- | --- |
| `draft` → `signed` | `current_user.role = client_user`; `current_user.client_id = ticket.client_id`; статус заявки ≠ `cancelled`; заявка существует |
| Сохранение позиций | `quantity > 0`; `unit_price ≥ 0`; если `item_type=service` то `service_id` обязателен; если `item_type=part` то `part_id` обязателен и `spare_parts.unit_price > 0` (BR-F-122) |
| Редактирование | `signed_by = NULL` (акт не подписан), иначе 403 Forbidden (BR-F-115) |
| Редактирование при существующем счёте | Если по заявке создан счёт (статус ≠ `cancelled`) — редактирование разрешено **только admin** (BR-F-126). Для остальных ролей — 403 `ACT_LOCKED_INVOICE_EXISTS`. |
| Сохранение акта при неоплаченном счёте | Если позиции акта изменились и счёт **не оплачен** — система автоматически пересчитывает позиции и итоги счёта (BR-F-127). |
| Сохранение акта при оплаченном счёте | Если позиции изменились и счёт **оплачен** — система возвращает 409 `INVOICE_PAID_MISMATCH` с полями `act_total` и `invoice_total`. Пользователь может сохранить акт принудительно (`force_save=true`); счёт при этом не изменяется (BR-F-127). |
| Создать счёт из акта | Акт содержит хотя бы одну позицию (`items` не пустой) (BR-F-118) |
