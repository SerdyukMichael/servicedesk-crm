# UC-1001: Создать / редактировать паспорт оборудования

**Версия:** 1.1 | **Дата:** 31.03.2026 | **Статус:** Реализовано

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-1001 |
| **Модуль** | Модуль 10: Учёт оборудования |
| **Основной актор** | Руководитель сервиса |
| **Второстепенные акторы** | Менеджер по продажам (авто-создание при продаже → BR-F-307), Система |
| **Связанные BR-F** | BR-F-1000, BR-F-1002, BR-F-1003 |
| **Связанные BR-R** | BR-R-008 (серийный номер уникален), BR-R-009 (soft delete), BR-R-010 (аудит-лог) |
| **Приоритет** | Must |
| **Предусловия** | Пользователь авторизован с ролью **Руководитель сервиса** или **Администратор**; клиент-владелец существует в системе; модель оборудования есть в справочнике |
| **Постусловия** | Паспорт оборудования создан / обновлён; гарантийный статус рассчитан автоматически |
| **Триггер** | Пользователь нажимает **«+ Добавить оборудование»** или открывает существующий паспорт для редактирования. Также: авто-создание при завершении продажи (BR-F-307) |

---

## Основной поток — Ручное создание

1. Пользователь открывает раздел **«Оборудование»** → нажимает **«+ Добавить»**.
2. Система отображает форму паспорта.
3. Пользователь заполняет обязательные поля: модель (из справочника), серийный номер, дата производства, дата продажи, клиент-владелец, адрес установки, дата начала и окончания гарантии.
4. Опционально заполняет конфигурацию: установленные модули (список из справочника), версию прошивки, комплектацию при поставке (BR-F-1003).
5. Нажимает **«Сохранить»**.
6. Система валидирует уникальность серийного номера (BR-R-008).
7. Система создаёт паспорт со статусом **«Активно»**.
8. Система автоматически рассчитывает гарантийный статус (BR-F-1002):
   - До окончания > 30 дней → 🟢 «На гарантии»
   - До окончания ≤ 30 дней → 🟠 «Гарантия истекает»
   - Гарантия истекла → 🔴 «Гарантия истекла»
9. Система фиксирует создание в аудит-логе.

---

## Основной поток — Авто-создание при продаже (BR-F-307)

1. Менеджер по продажам завершает отгрузку оборудования (→ Модуль 3), вводит серийный номер.
2. Система автоматически создаёт паспорт с данными из заказа: модель, серийный номер, клиент, дата продажи.
3. Дата начала гарантии = дата продажи. Срок гарантии — из карточки модели (справочник).
4. Паспорт создаётся со статусом **«Активно»**.
5. Руководитель сервиса дополняет паспорт (адрес установки, конфигурация) при необходимости.

---

## Исключения

**ИП-1: Серийный номер уже существует (BR-R-008)**
→ Система блокирует: *«Серийный номер [SN] уже существует: [Модель], клиент: [Название]»*.

**ИП-2: Дата окончания гарантии раньше даты начала**
→ *«Дата окончания гарантии должна быть позже даты начала»*.

## Acceptance Criteria

```gherkin
Feature: Паспорт оборудования

  Scenario: Успешное создание паспорта
    Given клиент «Банк Альфа», модель «Matica XID 580i», SN-99999 (уникальный)
    When Руководитель сервиса создаёт паспорт с гарантией до 01.01.2027

  Scenario: Блокировка при дубликате серийного номера
    Given SN-12345 уже зарегистрирован у клиента «Банк Бета»
    When создаётся паспорт с SN-12345
    Then система показывает «Серийный номер SN-12345 уже существует: клиент Банк Бета»

  Scenario: Авто-статус «Гарантия истекает»
    Given warranty_end = через 20 дней
    Then warranty_status = «Гарантия истекает» (оранжевый)

  Scenario: Авто-создание паспорта при продаже
    When менеджер завершает отгрузку банкомата «Matica XID 580i» SN-77777 клиенту «Банк Гамма»
    Then паспорт для SN-77777 создаётся автоматически
    And warranty_start = дата продажи
    And статус = «Активно»
```

## Примечания / Бизнес-правила

- **BR-R-008:** серийный номер уникален глобально. Система не позволяет сохранить дубликат.
- **BR-F-1002:** гарантийный статус пересчитывается автоматически (фоновый процесс раз в сутки + при каждом открытии карточки).
- Конфигурация оборудования (BR-F-1003) — дополнительный блок паспорта, не блокирует создание.

## Проектные решения (СА)

### Изменения схемы БД (таблица `equipment`)

Добавлены новые поля:

| Поле | Тип | По умолчанию | Обоснование |
|---|---|---|---|
| `manufacture_date` | DATE NULL | NULL | Дата производства оборудования |
| `sale_date` | DATE NULL | NULL | Дата продажи клиенту (используется для авто-создания паспорта) |
| `warranty_start` | DATE NULL | NULL | Начало гарантии; отделено от `installed_at` (дата установки) |
| `firmware_version` | VARCHAR(64) NULL | NULL | Версия прошивки для диагностических целей |

Существующие поля, используемые как алиасы:

| Поле в БД | Алиас в API | Назначение |
|---|---|---|
| `location` | `install_address`, `address` | Адрес установки |
| `installed_at` | `installation_date` | Дата установки |
| `warranty_until` | `warranty_end` | Дата окончания гарантии |

Статус enum расширен: добавлено значение `transferred` (передано другому клиенту).

### Вычисляемое поле `warranty_status`

Рассчитывается в Pydantic-схеме на основе `warranty_until` (хранится в БД) и текущей даты:

```
today > warranty_until              → "expired"
today >= warranty_until - 30 дней  → "expiring"
today < warranty_until - 30 дней   → "on_warranty"
warranty_until = NULL               → "unknown"
```

Не хранится в БД. Пересчитывается при каждом запросе.

### RBAC

Создание и редактирование: роли `admin`, `svc_mgr`.
Чтение: любой аутентифицированный пользователь.
Удаление (soft delete): только `admin`.
