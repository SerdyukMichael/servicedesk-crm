## Жизненный цикл заявки

```
Новая → Назначена → В работе → Ожидание запчасти → Выполнена → На проверке → Закрыта
                       ↑__________________|
```

| Переход | Кто может | Условия |
|---|---|---|
| Новая → Назначена | Руководитель сервиса | Назначен инженер (→ UC-902) |
| Назначена → В работе | Сервисный инженер, Руководитель сервиса | — |
| В работе → Ожидание запчасти | Сервисный инженер, Руководитель сервиса | — |
| Ожидание запчасти → В работе | Сервисный инженер, Руководитель сервиса | — |
| В работе → Выполнена | Сервисный инженер | Акт выполненных работ заполнен (→ UC-904) |
| Выполнена → На проверке | Руководитель сервиса | — |
| На проверке → Закрыта | Руководитель сервиса | Акт выполненных работ сохранён (BR-R-004) |
| На проверке → В работе | Руководитель сервиса | Возврат на доработку |

---

## Основной поток — Смена статуса

1. Пользователь открывает карточку заявки.
2. Система отображает текущий статус и доступные переходы (только разрешённые для роли пользователя).
3. Пользователь нажимает кнопку целевого статуса.
4. Система проверяет выполнение условий перехода (см. таблицу выше).
5. Система обновляет статус заявки.
6. Система фиксирует смену в аудит-логе: `action=status_change`, `old_status`, `new_status`, `user`, `timestamp` (BR-R-010).
7. Система обновляет статус оборудования автоматически:
   - При переходе в «В работе» → оборудование получает статус **«На ремонте»** (BR-F-1007)
   - При переходе в «Закрыта» → оборудование возвращается в статус **«Активно»**
8. Если новый статус из набора {Назначена, В работе, Выполнена, Закрыта} — система отправляет email клиенту (BR-F-910).

---

## Альтернативные потоки

**АП-1: Переход в «Ожидание запчасти»**
Инженер указывает, какой запчасти не хватает (выбор из справочника номенклатуры, опционально).
Статус меняется; SLA-таймер на решение продолжает работать (ожидание запчасти не останавливает SLA).

**АП-2: Возврат «На проверке» → «В работе»**
Руководитель сервиса указывает причину возврата (обязательный комментарий).
Инженер получает уведомление о возврате с текстом комментария.

---

## Исключения

**ИП-1: Переход в «Закрыта» без акта выполненных работ (BR-R-004)**
→ Система блокирует: *«Заполните акт выполненных работ перед закрытием заявки»*.
→ Кнопка «Закрыта» не активна до тех пор, пока акт не сохранён.

**ИП-2: Попытка недопустимого перехода**
Пользователь пытается перейти в статус, не предусмотренный из текущего (например, Новая → Закрыта).
→ Система блокирует: *«Недопустимый переход статуса»*. Кнопка недопустимого перехода не отображается.

**ИП-3: Переход недоступен для роли пользователя**
→ Кнопка перехода не отображается для роли, у которой нет прав.

## Acceptance Criteria

```gherkin
Feature: Жизненный цикл заявки

  Scenario: Успешный переход «Назначена» → «В работе»
    Given заявка #42 в статусе «Назначена», инженер «Петров» открывает её
    When «Петров» нажимает «Взять в работу»
    Then статус заявки = «В работе»
    And оборудование по заявке получает статус «На ремонте»
    And клиент получает email «Заявка взята в работу»
    And в аудит-логе: action=status_change, old=Назначена, new=В работе

  Scenario: Блокировка перехода «На проверке» → «Закрыта» без акта
    Given заявка в статусе «На проверке», акт выполненных работ не заполнен
    When Руководитель сервиса пытается перевести в «Закрыта»
    Then кнопка «Закрыта» не активна
    And при нажатии: «Заполните акт выполненных работ перед закрытием»

  Scenario: Переход «Закрыта» — оборудование возвращается в статус «Активно»
    Given заявка закрыта, оборудование SN-12345 было в статусе «На ремонте»
    When статус заявки становится «Закрыта»
    Then оборудование SN-12345 получает статус «Активно»

  Scenario: Возврат заявки на доработку
    Given заявка в статусе «На проверке»
    When Руководитель сервиса возвращает в «В работе» с комментарием «Не устранена первопричина»
    Then статус = «В работе»
    And инженер получает уведомление с текстом «Не устранена первопричина»
```

## Примечания / Бизнес-правила

- **BR-R-004:** закрытие без акта выполненных работ технически невозможно.
- **BR-R-010:** каждое изменение статуса фиксируется в аудит-логе без исключений.
- **BR-F-910:** уведомления клиенту — только при статусах Назначена, В работе, Выполнена, Закрыта. Переход в «Ожидание запчасти» или «На проверке» клиенту не отправляется.
- Переходы доступны только ролям, указанным в таблице жизненного цикла. Интерфейс скрывает недоступные кнопки (не блокирует серым, а не показывает).
