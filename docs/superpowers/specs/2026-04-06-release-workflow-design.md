# Release Workflow Design — ServiceDesk CRM

**Дата:** 2026-04-06  
**Версия:** 1.0  
**Статус:** Approved

---

## Контекст

Проект ServiceDesk CRM. До введения этого процесса не было формального «комплекта релиза» — непонятно что именно накатывается на боевой сервер, особенно по части миграций БД.

### Терминология
- **Локальный стенд** — Docker Compose на машине разработчика, разработка и тесты
- **Боевой сервер** — удалённый сервер 188.120.243.122, production

---

## Цели

1. Прозрачность — видеть что именно изменилось в каждой версии (Frontend / Backend / Database)
2. Контроль БД — SQL-дамп миграций для проверки до наката
3. История — по git-тегам можно вернуться к любой версии
4. Процесс — чёткий порядок шагов от разработки до деплоя

---

## Артефакты релиза

### Структура папки

```
releases/
└── vX.Y.Z/
    ├── CHANGES.md        # что изменилось: Frontend / Backend / Database
    └── db_migrations.sql # SQL-дамп миграций этой версии
```

### Шаблон CHANGES.md

```markdown
# vX.Y.Z — [краткое название]

## Frontend
- ...

## Backend
- ...

## Database
- Миграции: <revision_id>, <revision_id>
- Изменения: <описание схемы>

## Документы
- Обновлены: <список doc-файлов>
```

### Генерация db_migrations.sql

```bash
docker compose exec backend alembic upgrade head --sql > releases/vX.Y.Z/db_migrations.sql
```

Файл показывает точный SQL, который будет выполнен на боевом сервере — проверяется до наката.

## Пошаговый процесс релиза

```
1. РАЗРАБОТКА (разработчик + Claude, на локальном стенде)
   ├── Обновляем UC-документы, ER_DataModel, RBAC_Matrix по ходу работы
   ├── Пишем код (backend / frontend)
   └── Прогоняем тесты: docker compose exec backend pytest tests/ -v

2. ПРОВЕРКА (разработчик)
   ├── Проверка на локальном стенде — интерфейс, поведение, сценарии
   └── Фиксируем замечания → Claude правит → повтор

3. ФОРМИРОВАНИЕ РЕЛИЗА (Claude, по команде разработчика)
   ├── Создаёт releases/vX.Y.Z/CHANGES.md
   ├── Генерирует releases/vX.Y.Z/db_migrations.sql
   └── Показывает оба файла разработчику

4. ОДОБРЕНИЕ (разработчик)
   └── Читает CHANGES.md и db_migrations.sql → даёт добро или запрашивает правки

5. GIT (Claude, после одобрения)
   ├── git add + git commit -m "feat: vX.Y.Z — краткое название"
   ├── git tag vX.Y.Z
   └── git push && git push --tags

6. ДЕПЛОЙ НА БОЕВОЙ СЕРВЕР (Claude)
   ├── SSH → git pull → docker compose up -d --build
   └── docker compose exec backend alembic upgrade head
```

## Версионирование

Семантическое версионирование `vX.Y.Z`:

| Сегмент | Когда меняется |
|---------|---------------|
| X (major) | Кардинальные изменения архитектуры / breaking changes |
| Y (minor) | Новая функциональность (новый UC, новый модуль) |
| Z (patch) | Багфиксы, мелкие правки без новых фич |

Текущая версия: `v0.6.0`. Следующая minor: `v0.7.0`.

### Просмотр истории по версиям

```bash
git tag                                           # все версии
git diff v0.6.0 v0.7.0                           # diff между версиями
git show v0.7.0:releases/v0.7.0/CHANGES.md       # CHANGES конкретной версии
```

## Обновление документации

При каждом релизе обязательно обновлять (если затронуты):
- `docs/CHANGELOG.md` — добавить секцию версии
- `docs/ER_DataModel.md` — если изменилась схема БД
- `docs/RBAC_Matrix.md` — если изменились роли/права
- `docs/UC-xxx.md` — если изменилась бизнес-логика UC

---

## Что НЕ входит в процесс

- Staging-окружение (нет в текущей инфраструктуре)
- Автоматический rollback (делается вручную через `alembic downgrade -1`)
- Release branches / PR-flow (не нужен при текущем размере команды)
