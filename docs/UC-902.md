# UC-902: Назначить инженера на заявку

**Версия:** 1.0 | **Дата:** 26.03.2026 | **Статус:** На проверке

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-902 |
| **Модуль** | Модуль 9: Заявки на обслуживание |
| **Основной актор** | Руководитель сервиса |
| **Второстепенные акторы** | Система (SLA-таймер, push/email уведомление инженеру → BR-F-914) |
| **Связанные BR-F** | BR-F-804, BR-F-808, BR-F-903, BR-F-914 |
| **Связанные BR-R** | BR-R-010 (аудит-лог смены статуса) |
| **Приоритет** | Must |
| **Предусловия** | Заявка существует, статус — **«Новая»**; пользователь авторизован с ролью **Руководитель сервиса** или **Администратор** |
| **Постусловия** | Заявка переведена в статус **«Назначена»**; инженер получил push + email уведомление; зафиксировано время реакции (для SLA) |
| **Триггер** | Руководитель сервиса открывает заявку со статусом «Новая» и нажимает **«Назначить инженера»** |

## Основной поток

1. Руководитель сервиса открывает заявку (статус «Новая») и нажимает **«Назначить инженера»**.
2. Система отображает панель загрузки инженеров (→ UC-804) в режиме назначения:
   - Предварительный фильтр: только инженеры с активной компетенцией на модель оборудования из заявки (→ BR-F-804)
   - Инженеры в отпуске скрыты (если период отсутствия активен)
   - Сортировка: по текущей загрузке (наименьшая нагрузка — первые)
3. Руководитель сервиса выбирает инженера и нажимает **«Назначить»**.
4. Система переводит заявку в статус **«Назначена»**.
5. Система фиксирует `assigned_at = NOW()` — момент реакции для SLA.
6. Система фиксирует изменение статуса в аудит-логе (BR-R-010).
7. Система отправляет инженеру push-уведомление (≤ 1 минуты) и email (≤ 5 минут) с деталями заявки (→ BR-F-914).
8. Система отображает подтверждение: *«Заявка назначена на [ФИО инженера]»*.

---

## Альтернативные потоки

**АП-1: Инженер в отпуске — принудительное назначение**
Если Руководитель сервиса выбирает фильтр «Все» и видит инженера в отпуске:
→ Система показывает предупреждение (→ UC-805, АП-1).
→ При подтверждении назначение выполняется.

**АП-2: Переназначение (замена инженера)**
Заявка уже в статусе «Назначена» или «В работе».
Руководитель сервиса нажимает **«Переназначить»**.
Система показывает текущего инженера и предлагает выбрать нового.
После выбора: заявка остаётся в том же статусе; предыдущий инженер получает уведомление об отмене; новый — уведомление о назначении.
Аудит-лог фиксирует переназначение: `action=reassign, from=[old], to=[new]`.

---

## Исключения

**ИП-1: Нет доступных инженеров с нужной компетенцией**
→ Система показывает: *«Нет инженеров с действующей компетенцией на [Модель]. Проверьте профили компетенций (→ UC-803)»*.
→ Руководитель сервиса может расширить фильтр или переназначить вручную.

**ИП-2: SLA реакции уже нарушен на момент назначения**
→ Система сохраняет назначение, но помечает заявку флагом `sla_reaction_violated = true`.
→ В карточке заявки отображается: *«SLA реакции нарушен: назначено через [N ч. M мин.]»*.

---

## Ключевые данные

| Поле | Тип | Описание |
|---|---|---|
| assignee_id | FK → users | Назначенный инженер |
| assigned_at | timestamp | Момент назначения (фиксация времени реакции) |
| sla_reaction_violated | boolean | true, если assigned_at > sla_reaction_deadline |
| status | enum | → «Назначена» |

  Scenario: Переназначение инженера
    Given заявка #42 назначена на «Иванова»

    Then assignee_id = ID «Петрова»

    And «Петров» получает уведомление о назначении
    And статус заявки не меняется

## Примечания / Бизнес-правила

- **BR-F-804:** фильтрация инженеров по компетенции применяется автоматически. Принудительное назначение вне фильтра доступно только через явный выбор «Все».
- **BR-F-914:** уведомление инженеру — push ≤ 1 минуты + email ≤ 5 минут.
- **BR-R-010:** смена статуса на «Назначена» фиксируется в аудит-логе с полями: дата/время, пользователь, предыдущий статус, новый статус.
- Время реакции SLA считается от `created_at` заявки до `assigned_at` (момента назначения инженера).
