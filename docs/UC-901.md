# UC-901: Создать заявку на обслуживание

**Версия:** 1.4 | **Дата:** 26.04.2026 | **Статус:** В работе

> **v1.1:** добавлен поток самостоятельной регистрации заявки клиентом через веб-портал или мобильное приложение.
> **v1.2:** добавлен поток редактирования заявки после создания (П-6 аудита документации).
> **v1.3:** добавлен аудит-лог редактирования; список реквизитов расширен полями `client_stated_priority`, `client_desired_deadline`.
> **v1.4:** добавлено поле «Серийный номер» с автоопределением клиента, оборудования и типа заявки. Для `client_user`: клиент из сессии, оборудование фильтруется по организации. Новый API-эндпоинт `GET /equipment/lookup?serial=`.

### Состояния заявки

| Код | Название | Описание |
|---|---|---|
| `new` | Новая | Заявка создана, ожидает назначения инженера |
| `assigned` | Назначена | Инженер назначен, работы не начаты |
| `in_progress` | В работе | Инженер выполняет работы |
| `waiting_part` | Ожидание запчасти | Работы приостановлены — ждём запчасти |
| `on_review` | На проверке | Работы завершены, ожидается подтверждение |
| `completed` | Завершена | Работы подтверждены клиентом или системой |
| `closed` | Закрыта | Заявка закрыта, счёт выставлен |
| `cancelled` | Отменена | Заявка отменена до начала работ |

## Основной поток — Ручное создание

1. Руководитель сервиса нажимает **«+ Создать заявку»**.
2. Система отображает форму создания заявки.
3. Пользователь выбирает **клиента** из справочника.
4. Система подгружает список оборудования этого клиента (из Модуля 10).
5. Пользователь выбирает **оборудование** по серийному номеру.
6. Система автоматически определяет:
   - Гарантийный статус оборудования (→ BR-F-1002)
   - Договор обслуживания клиента (→ BR-F-1300) и его тип (Премиум / Стандарт)
   - Приоритет по умолчанию (→ BR-F-902): банкомат = высокий, карт-машина = средний; Премиум-договор повышает приоритет на один уровень
7. Пользователь выбирает **тип работ**: Гарантийный ремонт / Плановое ТО / Внеплановый ремонт / Установка.
8. Пользователь заполняет **описание проблемы** (минимум 10 символов).
9. Пользователь корректирует приоритет вручную при необходимости (с обязательным указанием причины).
10. Опционально: пользователь выбирает **шаблон типовых работ** (→ BR-F-913) — форма предзаполняется перечнем работ и запчастей.
11. Пользователь нажимает **«Создать заявку»**.
12. Система валидирует данные.
13. Система создаёт заявку (статус = **Новая**), запускает SLA-таймер (→ BR-F-903).
14. Система автоматически создаёт запись в истории ремонтов оборудования (BR-F-906).
15. Система отображает созданную заявку с уникальным номером.

## Дополнительный поток — Поиск по серийному номеру (v1.4)

1. На шаге 2 (или до выбора клиента) пользователь вводит **серийный номер** в поле «Серийный номер» (`serial_number_lookup`).
2. После ввода ≥ 3 символов система отправляет запрос `GET /api/v1/equipment/lookup?serial={sn}`.
3. **Оборудование найдено:**
   - Поля **«Клиент»** и **«Оборудование»** автоматически заполняются найденными значениями и **блокируются** для редактирования (иконка замка + тултип: *«Определено по серийному номеру»*).
   - Тип работ автоматически устанавливается **«Внеплановый ремонт»** (значение можно изменить вручную).
   - Система продолжает шаги 6–15 основного потока.
4. **Оборудование не найдено:** поле подсвечивается красным, сообщение: *«Оборудование с серийным номером [SN] не найдено»*. Поля клиент / оборудование остаются доступными для ручного ввода.
5. Пользователь может **очистить** поле серийного номера — тогда поля клиент и оборудование разблокируются.

**Для `client_user`:**
- Поле «Клиент» всегда предустановлено из сессии и заблокировано.
- Список оборудования фильтруется только по организации пользователя.
- Если введённый серийный номер принадлежит другой организации — ошибка: *«Данное оборудование не принадлежит вашей организации»*.

## Основной поток — Редактирование заявки

1. Руководитель сервиса открывает карточку заявки (статус — не «Закрыта», не «Отменена»).
2. Нажимает **«Редактировать»**.
3. Система переводит форму в режим редактирования. Доступные для изменения поля:

| Поле | Доступно при статусе |
|---|---|
| Заголовок | Любой (кроме Закрыта, Отменена) |
| Описание | Любой (кроме Закрыта, Отменена) |
| Тип работ | Только «Новая», «Назначена» |
| Приоритет | Любой (кроме Закрыта, Отменена); обязательна причина изменения |
| Контактное лицо | Любой (кроме Закрыта, Отменена) |

4. Поля, **недоступные** для редактирования: клиент, оборудование, инициатор, канал, номер заявки — заблокированы интерфейсом.
5. Пользователь вносит изменения → нажимает **«Сохранить»**.
6. Система сохраняет изменения, фиксирует в аудит-логе (`action: update`, изменённые поля со старыми и новыми значениями).
7. Система отображает **«Изменения сохранены»**.

**ИП — Редактирование типа работ после начала выполнения:**
При статусе «В работе» и выше — поле «Тип работ» заблокировано. Тултип: *«Тип работ нельзя изменить после начала выполнения»*.

---

## Основной поток — Автоматическое создание (Плановое ТО)

1. За 7 дней до даты планового ТО система автоматически создаёт заявку типа «Плановое ТО» (BR-F-909).
2. Заявка создаётся со статусом **«Новая»**, клиент и оборудование заполняются автоматически из графика ТО.
3. Система уведомляет Руководителя сервиса о созданной заявке (→ UC-1403).

## Дополнительный поток — Самостоятельная регистрация клиентом (веб-портал / мобильное приложение)

1. Сотрудник клиента открывает **веб-портал** или **мобильное приложение** и проходит аутентификацию.
2. Система автоматически определяет:
   - **Организацию клиента** — из сессии авторизованного пользователя (`client_id`).
   - **ФИО регистрирующего** — из профиля авторизованного пользователя (`initiator_id`, `initiator_name`).
   - **Канал** — `web_form` или `mobile_app` (`channel`).
3. Клиент может ввести **серийный номер** → система автоматически определяет оборудование (только из оборудования своей организации); тип устанавливается «Внеплановый ремонт».
4. Клиент выбирает **причину обращения**:
   - «Поломка / неисправность» → `work_type = unplanned_repair`
   - «Сервисное обслуживание» → `work_type = planned_maintenance`
5. Клиент выбирает **оборудование** из списка зарегистрированного оборудования своей организации (по серийному номеру / модели).
6. Система автоматически определяет гарантийный статус и договор (те же правила, что в шагах 6 основного потока).
7. Клиент заполняет **описание проблемы** (опционально, но рекомендуется; минимум 10 символов если заполнено).
8. Клиент **прикладывает файлы** (фото, видео, скриншоты) — опционально. Максимум 10 файлов, до 20 МБ каждый.
9. Клиент указывает (всё опционально):
   - **Контактное лицо** по данной заявке — имя и телефон (может отличаться от регистрирующего).
   - **Желаемый срок устранения** (`client_desired_deadline`) — дата.
   - **Критичность для клиента** (`client_stated_priority`): Низкая / Средняя / Высокая / Критичная.
10. Клиент нажимает **«Отправить заявку»**.
11. Система валидирует данные (обязательные поля: организация, оборудование, причина обращения).
12. Система создаёт заявку:
    - Статус = **«Новая»**
    - `initiator_type = client`, `initiator_id` = ID авторизованного сотрудника клиента
    - `channel` = `web_form` или `mobile_app`
    - `created_by_user_id` = system_user
    - SLA-таймер запускается (→ BR-F-903)
    - Запись в истории ремонтов оборудования (→ BR-F-906)
13. Система уведомляет **Руководителя сервиса** о новой заявке от клиента (→ UC-1403).
14. Клиенту отображается подтверждение с номером заявки: *«Ваша заявка № [SD-YYYY-NNNNNN] принята. Ожидайте звонка в течение [SLA реакции]»*.

> **Примечание по приоритету:** `client_stated_priority` учитывается как дополнительный фактор при автоматическом расчёте системного приоритета (BR-F-902). Итоговый приоритет устанавливается системой — Руководитель сервиса может его скорректировать при обработке заявки.

## Альтернативные потоки

**АП-1: Гарантийный ремонт**
На шаге 7 выбран тип «Гарантийный ремонт».
Система проверяет гарантийный статус оборудования:
- Если «На гарантии» — устанавливает флаг `is_warranty = true`. Заявка не тарифицируется клиенту (BR-R-006).
- Если «Гарантия истекла» — система предупреждает: *«Гарантия на оборудование истекла [дата]. Продолжить как гарантийный?»*. Пользователь подтверждает или меняет тип.

**АП-2: Применение шаблона типовых работ**
На шаге 10 пользователь выбирает шаблон (→ BR-F-913).
Система предзаполняет: перечень работ, список необходимых запчастей с количеством.
Пользователь может скорректировать предзаполненные данные перед сохранением.

## Исключения

**ИП-1: Описание проблемы короче 10 символов**
→ Система блокирует: *«Описание проблемы должно содержать минимум 10 символов»*.

**ИП-2: У клиента нет зарегистрированного оборудования**
→ Система показывает: *«У клиента нет оборудования. Сначала зарегистрируйте оборудование в Модуле 10»*.

**ИП-3: Нет действующего договора обслуживания**
→ Система предупреждает: *«У клиента нет активного договора обслуживания. SLA будет применён по умолчанию (Стандарт)»*. Заявка создаётся.

**ИП-4: У клиента нет зарегистрированного оборудования (клиентский портал)**
На шаге 4 клиентского потока: список оборудования пуст.
→ Система показывает: *«В вашей организации нет зарегистрированного оборудования. Обратитесь к менеджеру»*. Отправка заявки блокируется.

**ИП-5: Клиент не идентифицирован**
Сессия истекла или пользователь не авторизован.
→ Система перенаправляет на страницу входа. Заявка не создаётся.

**ИП-6: Описание заполнено, но короче 10 символов (клиентский портал)**
→ Система блокирует: *«Описание должно содержать минимум 10 символов или оставьте поле пустым»*.

**ИП-7: Серийный номер не найден (v1.4)**
→ Поле серийного номера подсвечивается красным: *«Оборудование с серийным номером [SN] не найдено»*. Поля клиент и оборудование остаются доступными для ручного заполнения.

**ИП-8: Серийный номер принадлежит другой организации (client_user, v1.4)**
→ Ошибка: *«Данное оборудование не принадлежит вашей организации»*. Автозаполнение не происходит.

## API-эндпоинт: Поиск оборудования по серийному номеру (v1.4)

**BR-F-918:** Система должна предоставлять эндпоинт поиска оборудования по серийному номеру для автозаполнения формы заявки.

```
GET /api/v1/equipment/lookup?serial={serial_number}
Authorization: Bearer {token}
```

**Ответ (200 OK — найдено):**
```json
{
  "equipment_id": 42,
  "serial_number": "SN-12345",
  "model_name": "Matica XID 580i",
  "client_id": 7,
  "client_name": "Банк Альфа",
  "is_under_warranty": true,
  "warranty_expiry": "2027-01-15"
}
```

**Ответ (404 Not Found — не найдено):**
```json
{ "detail": "Equipment with serial number 'SN-99999' not found" }
```

**Поведение для `client_user`:**
- Если найденное оборудование принадлежит другой организации → `403 Forbidden`: `{"detail": "Equipment does not belong to your organization"}`.

**Поведение при `serial` пустой или < 3 символов:** `400 Bad Request`.

## Ограничения доступа для `client_user`

Пользователь с ролью `client_user` видит **только заявки своей организации** (`tickets.client_id == users.client_id`). Ограничение применяется как к списку, так и к прямому переходу по URL карточки заявки.

| Эндпоинт | Поведение для `client_user` |
|---|---|
| `GET /tickets` | возвращает только заявки своей организации |
| `GET /tickets/{id}` | HTTP 404, если `client_id` не совпадает с org пользователя |
| `GET /tickets/{id}/comments` | HTTP 404, если заявка чужой org; внутренние комментарии скрыты |
| `GET /tickets/{id}/attachments` | HTTP 404, если заявка чужой org |
| `GET /tickets/{id}/work-act` | HTTP 404, если заявка чужой org |
| `GET /tickets/{id}/status-history` | HTTP 404, если заявка чужой org |
| `POST /tickets/{id}/comments` | HTTP 404, если заявка чужой org |
| `POST /tickets/{id}/work-act/*/sign` | HTTP 403, если заявка чужой org |
| `GET /equipment/lookup?serial=` | HTTP 403, если оборудование чужой org |

> Намеренный выбор HTTP 404 (а не 403) для GET-эндпоинтов: утечка информации о существовании записи нежелательна. Для подписи акта и lookup чужого оборудования используется 403, так как факт существования заявки/оборудования пользователю уже известен.

## Примечания / Бизнес-правила

- **BR-F-902:** автоматический приоритет = f(тип договора, тип оборудования). Ручное изменение возможно с указанием причины.
- **BR-F-903:** SLA-таймер стартует в момент создания заявки (не назначения инженера).
- **BR-R-006:** гарантийная заявка не формирует счёт клиенту; стоимость работ и запчастей идёт в рекламацию вендору (→ Модуль 11).
- **BR-F-906:** запись в истории ремонтов оборудования создаётся автоматически при создании заявки и обновляется при её закрытии.
- **BR-F-916:** поля `initiator_type`, `initiator_id` и `channel` обязательны при создании заявки любым способом. В клиентском потоке заполняются автоматически из сессии.
- **BR-F-917:** при регистрации через портал или приложение `initiator_type = client`, `initiator_id` = ID авторизованного сотрудника клиента. `created_by_user_id = system_user`.
- **BR-F-918 (v1.4):** эндпоинт `GET /equipment/lookup?serial=` доступен всем авторизованным пользователям. Для `client_user` фильтрует по организации из сессии — оборудование другой org возвращает 403.
- `description` обязательно только при создании оператором (≥ 10 символов). При клиентской регистрации — опционально: если заполнено, проверяется минимум 10 символов.
- `client_stated_priority` — субъективная оценка клиента, не заменяет системный приоритет. Используется Руководителем сервиса как дополнительный контекст при обработке заявки.
- `serial_number_lookup` — поле только формы, не хранится в БД. Debounce на стороне фронта: запрос к `/equipment/lookup` — через 400 мс после последнего нажатия клавиши.
