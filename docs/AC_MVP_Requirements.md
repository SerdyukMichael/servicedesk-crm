### BR-R-001 — Выставление счёта: только Менеджер или Бухгалтер

```gherkin
Feature: BR-R-001 Права на выставление счёта

  Scenario: Менеджер по продажам создаёт счёт — успешно
    Given пользователь авторизован с ролью «Менеджер по продажам»
    When он создаёт счёт на основе заказа #ORD-001
    Then счёт сохранён со статусом «Черновик»
    And в истории счёта зафиксирован создатель

  Scenario: Бухгалтер создаёт счёт — успешно
    Given пользователь авторизован с ролью «Бухгалтер»
    When он создаёт счёт на основе заявки на обслуживание #SR-042
    Then счёт сохранён

  Scenario: Сервисный инженер пытается создать счёт — блокировка
    Given пользователь авторизован с ролью «Сервисный инженер»
    When он пытается перейти в раздел «Счета» и создать счёт
    Then система возвращает ошибку доступа 403
    And кнопка «Создать счёт» не отображается в интерфейсе

  Scenario: Директор пытается создать счёт — блокировка
    Given пользователь авторизован с ролью «Директор»
    When он пытается создать счёт напрямую
    Then система блокирует: кнопка недоступна (только просмотр)
```

### BR-R-002 — Скидка > 15% на позицию: требует согласования Руководителя отдела продаж

```gherkin
Feature: BR-R-002 Контроль скидки на позицию заказа

  Scenario: Скидка 10% — сохраняется без согласования
    Given Менеджер по продажам создаёт заказ
    When он устанавливает скидку 10% на позицию «Картридж XID»
    And нажимает «Сохранить»
    Then заказ сохранён без запроса согласования

  Scenario: Скидка 20% — требует согласования Руководителя продаж
    Given Менеджер по продажам устанавливает скидку 20% на позицию
    When он нажимает «Сохранить»
    Then система блокирует сохранение
    And создаётся запрос на согласование Руководителю отдела продаж
    And заказ переходит в статус «Ожидание согласования»

  Scenario: Руководитель отдела продаж согласовывает скидку 20%
    Given заказ в статусе «Ожидание согласования» со скидкой 20%
    When Руководитель отдела продаж нажимает «Согласовать»
    Then заказ переходит в статус «Черновик» (редактируемый)
    And скидка 20% зафиксирована
```

### BR-R-003 — Скидка > 25% на общую сумму заказа: требует согласования Директора

```gherkin

  Scenario: Директор согласовывает скидку 30%
    Given у Директора есть запрос на согласование
    When Директор нажимает «Согласовать»
    Then заказ сохраняется со скидкой 30%
    And в истории заказа зафиксировано согласование Директора
```

### BR-R-004 — Закрытие заявки без акта выполненных работ запрещено

```gherkin
Feature: BR-R-004 Обязательность акта выполненных работ

  Scenario: Закрытие заявки с заполненным актом — успешно
    Given заявка #42 в статусе «На проверке»
    And акт выполненных работ сохранён с описанием, фото и client_confirmed = true
    When Руководитель сервиса нажимает «Закрыта»
    Then статус заявки = «Закрыта»

  Scenario: Кнопка «Закрыта» неактивна при отсутствии акта
    Given заявка #43 в статусе «На проверке»
    And акт выполненных работ не заполнен
    When Руководитель сервиса открывает карточку заявки
    Then кнопка «Закрыта» не отображается (скрыта, не серая)

  Scenario: API-попытка перевести заявку в «Закрыта» без акта — отклонение
    Given заявка #44 без акта выполненных работ
    When отправляется PATCH /api/v1/requests/44 с body {"status": "closed"}
    Then API возвращает 422 Unprocessable Entity
    And тело ответа содержит: "Акт выполненных работ обязателен для закрытия"
```

### BR-R-005 — Списание запчастей только с привязкой к заявке или заказу

```gherkin
Feature: BR-R-005 Привязка списания запчастей к основанию

  Scenario: Списание запчасти через акт выполненных работ — успешно
    Given инженер заполняет акт по заявке #42
    When он добавляет «Картридж XID» 1 шт.
    And сохраняет акт
    Then остаток «Картридж XID» уменьшается на 1
    And запись расхода содержит ticket_id = 42

  Scenario: Списание через заказ на продажу — успешно
    Given Менеджер создаёт заказ на продажу #ORD-010
    When он указывает «Картридж XID» 2 шт. к отгрузке
    Then создаётся запись расхода с order_id = ORD-010

  Scenario: Попытка списания без основания через API — блокировка
    Given запрос POST /api/v1/parts/deduction без ticket_id и order_id
    When отправляется запрос
    Then API возвращает 422
    And сообщение: "Списание запрещено без привязки к заявке или заказу"
    And остаток на складе не изменяется

  Scenario: Акт инвентаризации — единственное исключение
    Given Кладовщик создаёт акт списания по результатам инвентаризации
    When указывает позицию и причину «Инвентаризация»
    Then списание выполняется с основанием act_type = inventory
```

### BR-R-006 — Гарантийный ремонт не тарифицируется клиенту

```gherkin
Feature: BR-R-006 Гарантийный ремонт — без тарификации клиенту

  Scenario: Гарантийная заявка — счёт клиенту не выставляется
    Given оборудование SN-12345 находится на гарантии (гарантия до 31.12.2026)
    And создана заявка #50 с is_warranty = true
    When инженер закрывает заявку с актом
    Then счёт клиенту НЕ создаётся автоматически
    And система фиксирует стоимость работ как расходы (cost_type = warranty_expense)

  Scenario: Гарантийная заявка — данные акта передаются в рекламацию вендору
    Given закрыта гарантийная заявка #50 с использованными запчастями
    When система обрабатывает закрытие
    Then в модуле рекламаций создаётся черновик претензии вендору
    And черновик содержит: заявку #50, список запчастей, описание работ

  Scenario: Негарантийная заявка — стандартная тарификация
    Given оборудование SN-99999 без активной гарантии
    And создана заявка #51 с is_warranty = false
    When инженер закрывает заявку
    Then система позволяет выставить счёт клиенту на основе акта
```

### BR-R-007 — Заявка на закупку > 5000 EUR: согласование Директора

```gherkin
Feature: BR-R-007 Контроль суммы заявки на закупку

  Scenario: Заявка на закупку 3000 EUR — без согласования Директора
    Given Руководитель сервиса создаёт заявку на закупку запчастей на сумму 3000 EUR
    When нажимает «Отправить на утверждение»
    Then заявка направляется на утверждение Руководителю сервиса (стандартный поток)
    And Директор не уведомляется

  Scenario: Заявка на закупку 6000 EUR — требует согласования Директора
    Given создаётся заявка на закупку на сумму 6000 EUR (по курсу NBP на дату создания)
    When заявка отправляется на утверждение
    Then заявка направляется Директору на согласование
    And Директор получает уведомление email + in-app
    And статус заявки = «Ожидает согласования Директора»

  Scenario: Курс NBP рассчитывается автоматически
    Given создаётся заявка в PLN на сумму 25 000 PLN
    And курс EUR/PLN по NBP на текущую дату = 4.20
    Then система рассчитывает эквивалент = 25000 / 4.20 ≈ 5952 EUR
    And запрос направляется на согласование Директора
```

### BR-R-008 — Серийный номер оборудования глобально уникален

```gherkin
Feature: BR-R-008 Уникальность серийного номера оборудования

  Scenario: Регистрация оборудования с уникальным серийным номером
    Given в системе нет оборудования с серийным номером «SN-XYZ-001»
    When Руководитель сервиса создаёт паспорт оборудования с SN = «SN-XYZ-001»
    Then оборудование сохранено
    And паспорт привязан к клиенту «Банк Альфа»

  Scenario: Попытка зарегистрировать дублирующийся серийный номер — блокировка
    Given в системе уже есть оборудование SN = «SN-XYZ-001», клиент «Банк Альфа»
    When создаётся новое оборудование с SN = «SN-XYZ-001»
    Then система блокирует: «Серийный номер уже существует: SN-XYZ-001, клиент: Банк Альфа»
    And новая запись не сохраняется

  Scenario: Уникальность проверяется на уровне БД
    Given два одновременных POST /api/v1/equipment с одним SN
    Then только один запрос успешен (HTTP 201)
    And второй получает HTTP 409 Conflict
```

### BR-R-009 — Soft delete: физическое удаление запрещено во всех модулях

```gherkin
Feature: BR-R-009 Soft delete во всех модулях

  Scenario: Удаление сотрудника — soft delete
    Given в системе есть сотрудник «Иванов» (id=10)
    When Администратор нажимает «Уволить» и подтверждает
    Then запись пользователя получает is_deleted = true, deleted_at = текущее время
    And «Иванов» не отображается в списке сотрудников
    And «Иванов» не доступен для назначения на заявки

  Scenario: Удалённая запись сохраняется в аудит-логе
    Given сотрудник «Иванов» помечен is_deleted = true
    When Администратор открывает аудит-лог с фильтром entity=user, id=10
    Then все действия «Иванова» доступны в логе
    And запись удаления зафиксирована: action=delete, entity=user

  Scenario: API DELETE не выполняет физическое удаление
    Given сотрудник id=10 существует в БД
    When отправляется DELETE /api/v1/users/10
    Then запись в таблице users НЕ удаляется физически
    And is_deleted = true, deleted_at заполнен
    And ответ API: HTTP 200 с {"status": "deactivated"}

  Scenario: Soft delete применяется к комментариям
    Given комментарий id=55 существует для заявки #42
    When Администратор удаляет комментарий с причиной «Нарушение политики»
    Then comment.is_deleted = true
    And комментарий не отображается в ленте
    And в аудит-логе: action=delete, entity=comment, reason="Нарушение политики"

  Scenario: Soft delete применяется к оборудованию (списание)
    Given оборудование SN-12345 переводится в статус «Списано»
    Then equipment.is_deleted = true (или status = written_off)
    And оборудование не появляется в активных списках клиента
```

### BR-R-010 — Аудит-лог каждой смены статуса заявки

```gherkin
Feature: BR-R-010 Аудит-лог изменений статуса заявки

  Scenario: Переход статуса фиксируется в аудит-логе
    Given заявка #42 в статусе «Назначена»
    When инженер «Петров» нажимает «Взять в работу»
    Then в аудит-логе появляется запись:
      | action      | status_change     |
      | entity      | ticket            |
      | entity_id   | 42                |
      | old_status  | assigned          |
      | new_status  | in_progress       |
      | actor_id    | [ID Петрова]      |
      | created_at  | [текущий timestamp] |

  Scenario: Автоматическая смена статуса тоже логируется
    Given система автоматически эскалирует заявку #43
    When статус меняется системой (actor = system)
    Then в аудит-логе: actor_id = NULL, actor_type = system

  Scenario: Аудит-лог нельзя изменить или удалить
    Given в аудит-логе есть запись id=1001
    When попытка PATCH /api/v1/audit/1001
    Then API возвращает 405 Method Not Allowed
    When попытка DELETE /api/v1/audit/1001
    Then API возвращает 405 Method Not Allowed

  Scenario: Аудит-лог хранится минимум 2 года (BR-NF-014)
    Given запись в аудит-логе создана 2 года назад
    When запускается плановая очистка старых данных
    Then запись НЕ удаляется (хранение ≥ 730 дней)
    And записи старше 2 лет могут архивироваться, но не удаляться
```

### Модуль 8 — Учёт персонала

```gherkin
Feature: BR-F-800 Карточка сотрудника

  Scenario: Создание карточки сотрудника с обязательными полями
    Given Администратор открывает форму нового сотрудника
    When заполняет: ФИО «Иванов Иван Иванович», роль «Сервисный инженер»,
         email «ivanov@company.com», дата приёма 01.03.2026
    And нажимает «Сохранить»
    Then карточка сотрудника создана, статус = «Активен»
    And в аудит-логе: action=create, entity=user

  Scenario: Увольнение сотрудника
    Given сотрудник «Иванов» активен, назначен на 0 открытых заявок
    When Администратор нажимает «Уволить» и подтверждает
    Then статус сотрудника = «Уволен»
    And «Иванов» не отображается в списке при назначении заявок

Feature: BR-F-801 Ролевая модель доступа (RBAC)

  Scenario: Сервисный инженер не видит финансовые модули
    Given пользователь авторизован как «Сервисный инженер»
    When он открывает навигацию системы
    Then разделы «Счета», «Финансы», «Закупки» не отображаются

  Scenario: Директор имеет доступ только для чтения во всех модулях
    Given пользователь авторизован как «Директор»
    When он открывает карточку заявки
    Then кнопки «Назначить инженера», «Изменить статус» не отображаются
    And данные доступны в режиме просмотра

Feature: BR-F-802 Профиль компетенций инженера

  Scenario: Добавление компетенции без сертификата
    Given Руководитель сервиса открыл профиль инженера «Петров»
    When добавляет модель «Matica XID 580i» без заполнения сертификата
    Then компетенция сохранена
    And «Петров» отображается при назначении заявок на «Matica XID 580i»

  Scenario: Инженер с истёкшим сертификатом не отображается при назначении
    Given у инженера «Сидоров» certificate_valid_until = вчерашняя дата
    When открывается список инженеров для заявки на «Matica XID 580i»
    Then «Сидоров» отсутствует в списке доступных инженеров

Feature: BR-F-803 Панель загрузки инженеров

  Scenario: Панель отображает актуальные счётчики
    Given инженер «Кузнецов» имеет 5 заявок in_progress, 2 из них с нарушением SLA
    When Руководитель сервиса открывает панель загрузки
    Then строка «Кузнецов»: «Заявок в работе» = 5, «Просрочено» = 2
    And индикатор нагрузки = 🟡 (средняя, 4–6 заявок)

Feature: BR-F-806 Управление ролями (RBAC)

  Scenario: Администратор назначает роль пользователю
    Given пользователь «Новиков» создан без роли
    When Администратор назначает роль «Сервисный инженер»
    Then «Новиков» получает доступ к модулям инженера
    And в аудит-логе: action=roles_assign, entity=user

  Scenario: Пользователь без роли не может войти в систему
    Given создан пользователь без назначенной роли
    When он пытается авторизоваться
    Then система возвращает ошибку: «Аккаунт не активирован. Обратитесь к администратору»

Feature: BR-F-807 Аудит-лог

  Scenario: Аудит-лог фиксирует создание нового сотрудника
    Given Администратор создал сотрудника «Морозов»
    When Администратор открывает аудит-лог, фильтр: entity=user, action=create
    Then в логе присутствует запись с entity_id = [ID Морозова], actor = Администратор

  Scenario: Аудит-лог недоступен для роли «Сервисный инженер»
    Given пользователь авторизован как «Сервисный инженер»
    When GET /api/v1/audit
    Then API возвращает 403 Forbidden
```

### Модуль 9 — Заявки на обслуживание

```gherkin
Feature: BR-F-900 Регистрация заявки

  Scenario: Создание заявки сотрудником (основной поток)
    Given Руководитель сервиса авторизован
    When создаёт заявку: клиент «Банк Альфа», оборудование SN-001,
         тип «unplanned_repair», описание «Не работает картридер»
    Then заявка сохранена со статусом «Новая»
    And SLA-таймер запущен с момента created_at
    And initiator_type = employee, channel = web_form

  Scenario: Самостоятельная регистрация клиентом через портал
    Given контактное лицо клиента авторизовано на портале
    When регистрирует заявку: выбирает оборудование SN-001, причина «Поломка»
    Then заявка создана с initiator_type = client, channel = web_form
    And Руководитель сервиса получает уведомление о новой заявке

Feature: BR-F-901 Жизненный цикл заявки

  Scenario: Полный цикл заявки (happy path)
    Given заявка #42 в статусе «Новая»
    When Руководитель сервиса назначает инженера «Петров» → статус «Назначена»
    And «Петров» берёт в работу → статус «В работе»
    And «Петров» заполняет акт с client_confirmed = true → статус «Выполнена»
    And Руководитель переводит в «На проверке»
    And Руководитель переводит в «Закрыта» (акт есть)
    Then итоговый статус = «Закрыта»
    And в аудит-логе 6 записей action=status_change для заявки #42

  Scenario: Недопустимый переход статуса заблокирован
    Given заявка #43 в статусе «Новая»
    When попытка перевести в «Закрыта» напрямую через API
    Then API возвращает 422
    And сообщение: «Недопустимый переход статуса: Новая → Закрыта»

Feature: BR-F-903 SLA-контроль

  Scenario: SLA-таймер запускается при создании заявки
    Given создана заявка #44 в 10:00
    And SLA для клиента: реакция 4 часа, решение 24 часа
```
