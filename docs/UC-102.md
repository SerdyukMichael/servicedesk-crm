# UC-102: Управлять ценами материального каталога

**Версия:** 2.0 | **Дата:** 14.04.2026 | **Статус:** В работе

> **v2.0 (14.04.2026):** Полная переработка. Справочник `product_catalog` упразднён.
> UC-102 теперь описывает управление ценами позиций единого каталога матценностей (`spare_parts`).
> История цен ведётся через `price_history` как для услуг (`service_catalog`), так и для матценностей.

## Контекст: как устроен каталог

`spare_parts` — единый справочник материальных ценностей (запчасти, расходники, прочие товары). Каждая позиция имеет два аспекта:

- **Складской:** `quantity`, `min_quantity`, `vendor_id` — учёт наличия
- **Ценовой:** `unit_price`, `currency` — кэш актуальной цены; история изменений хранится в `price_history`

Не у каждой позиции есть цена (`unit_price = 0`). В форме акта и счёта отображаются **только позиции с установленной ценой** (`unit_price > 0`, BR-F-122).

---

## Основной поток: Установка / изменение цены

1. Пользователь открывает раздел **«Прайс-листы → Матценности»**.
2. Система отображает список позиций `spare_parts`:
   - Колонки: Артикул (SKU), Наименование, Категория, Ед. изм., Остаток, Текущая цена, Валюта, Статус цены
   - Статус цены: **«Цена установлена»** (`unit_price > 0`) / **«Без цены»** (`unit_price = 0`)
3. Пользователь нажимает **«Установить цену»** (если цена не задана) или **«Изменить цену»** (если уже есть).
4. Система открывает форму:
   - **Новая цена** — числовое поле, ≥ 0
   - **Валюта** — определяется системной настройкой (UC-1601); доступные значения: RUB, KZT, PLN и другие ISO 4217 коды
   - **Причина изменения** — текстовое поле, обязательно ≥ 5 символов
5. Пользователь заполняет форму и нажимает **«Сохранить»**.
6. Система валидирует: цена ≥ 0, причина заполнена.
7. Система:
   - Создаёт запись в `price_history` (`entity_type='spare_part'`, `entity_id`, `old_price`, `new_price`, `currency`, `reason`, `changed_by`, `changed_at`)
   - Обновляет `spare_parts.unit_price` и `spare_parts.currency`
8. Позиция отображается со статусом **«Цена установлена»** и обновлённым значением.

---

## Альтернативные потоки

**АП-1: Просмотр истории цен позиции**
Пользователь нажимает «История цен» у нужной позиции.
Система отображает список записей из `price_history`: дата, автор, старая цена, новая цена, валюта, причина.
Записи отсортированы от новых к старым. Редактирование и удаление недоступны (append-only).

**АП-2: Сброс цены (обнуление)**
Пользователь устанавливает цену `0.00` с указанием причины.
Система записывает в `price_history` и выставляет `unit_price = 0`.
Позиция получает статус **«Без цены»** и перестаёт отображаться в формах актов и счетов.

**АП-3: Фильтрация по статусу цены**
Пользователь применяет фильтр «Только с ценой» / «Только без цены» / «Все».
Система фильтрует список позиций по признаку `unit_price > 0`.

---

## Исключения

**ИП-1: Цена отрицательная**
→ Система блокирует: «Цена не может быть отрицательной».

**ИП-2: Причина изменения не указана или слишком короткая**
→ Система блокирует: «Укажите причину изменения цены (не менее 5 символов)».

**ИП-3: Позиция деактивирована (`is_active = false`)**
→ Изменение цены технически возможно, но деактивированная позиция всё равно не отображается в формах актов и счетов.

## Ключевые данные

### Форма изменения цены

| Поле | Обязательное | Тип | Ограничения |
| --- | --- | --- | --- |
| new_price | Да | decimal(12,2) | ≥ 0 |
| currency | Да | varchar(3) | Из системной настройки (UC-1601); допустимые коды: RUB, KZT, PLN и др. |
| reason | Да | varchar(512) | ≥ 5 символов |

### Запись price_history (создаётся автоматически)

| Поле | Значение |
| --- | --- |
| entity_type | `spare_part` |
| entity_id | `spare_parts.id` изменяемой позиции |
| old_price | `spare_parts.unit_price` до изменения |
| new_price | новая цена |
| currency | валюта |
| reason | причина |
| changed_by | ID текущего пользователя |
| changed_at | текущий timestamp |

## Acceptance Criteria

```gherkin
Feature: Управление ценами материального каталога

  Scenario: Успешная установка цены для позиции без цены
    Given пользователь с ролью svc_mgr открыл страницу «Прайс-листы → Матценности»
    And позиция SKU=PART-001 «Ролик захвата» имеет unit_price = 0
    When нажимает «Установить цену» для PART-001
    And вводит цену=1500, валюта=[системная валюта], причина=«Первичная установка цены»
    And нажимает «Сохранить»
    Then spare_parts.unit_price для PART-001 = 1500.00
    And в price_history создана запись: entity_type=spare_part, new_price=1500, old_price=0
    And позиция отображается со статусом «Цена установлена»

  Scenario: Успешное изменение цены с историей
    Given позиция SKU=PART-002 имеет unit_price = 800
    When пользователь вводит новую цену=950, причина=«Пересмотр прайса Q2»
    Then spare_parts.unit_price = 950.00
    And в price_history создана запись: old_price=800, new_price=950
    And в истории цен PART-002 отображаются обе записи

  Scenario: Позиция без цены не отображается в форме акта
    Given позиция SKU=PART-003 имеет unit_price = 0
    When инженер открывает форму создания акта и выбирает тип позиции «Запчасть»
    Then SKU=PART-003 отсутствует в выпадающем списке

  Scenario: Позиция с ценой отображается в форме акта
    Given позиция SKU=PART-004 имеет unit_price = 2000
    When инженер открывает форму создания акта и выбирает тип позиции «Запчасть»
    Then SKU=PART-004 присутствует в выпадающем списке с ценой 2 000 [системная валюта]

  Scenario: Блокировка при пустой причине
    Given пользователь открыл форму изменения цены
    When вводит новую цену=1200 и оставляет поле «Причина» пустым
    And нажимает «Сохранить»
    Then система показывает «Укажите причину изменения цены (не менее 5 символов)»
    And сохранение блокируется
```
