# UC-912: Отменить заявку

**Версия:** 1.0 | **Дата:** 31.03.2026 | **Статус:** Черновик

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-912 |
| **Модуль** | Модуль 9: Заявки на обслуживание |
| **Основной актор** | Руководитель сервиса |
| **Второстепенные акторы** | Администратор (те же права), Система (уведомление инженера, обновление статуса оборудования) |
| **Связанные BR-F** | BR-F-901 |
| **Связанные BR-R** | BR-R-004 (акт не нужен при отмене), BR-R-009 (soft delete не применяется — заявка меняет статус, не удаляется), BR-R-010 (аудит-лог) |
| **Приоритет** | Must |
| **Предусловия** | Заявка существует; статус заявки — не **«Закрыта»** и не **«Отменена»**; пользователь авторизован с ролью **Руководитель сервиса** или **Администратор** |
| **Постусловия** | Заявка переведена в статус **«Отменена»**; SLA-таймер остановлен; если был назначен инженер — инженер уведомлён; если оборудование было в статусе «На ремонте» — статус возвращается в «Активно» |
| **Триггер** | Руководитель сервиса открывает карточку заявки → нажимает **«Отменить заявку»** |

> **Обоснование добавления UC:** статус `cancelled` присутствует в ER-модели (`tickets.status`) и в жизненном цикле заявок, но ни UC-901, ни UC-903 не описывают сценарий и условия отмены. Без явного UC это поведение системы не определено.

---

## Контекст: разрешённые переходы в статус «Отменена»

| Текущий статус | Отмена разрешена? | Условие |
|---|---|---|
| Новая | Да | Без ограничений |
| Назначена | Да | С предупреждением о назначенном инженере |
| В работе | Да | С явным подтверждением + обязательная причина |
| Ожидание запчасти | Да | С явным подтверждением + обязательная причина |
| Выполнена | Нет | Акт уже заполнен, заявка ждёт проверки |
| На проверке | Нет | Аналогично — работы завершены |
| Закрыта | Нет | Финальный статус |
| Отменена | Нет | Уже отменена |

## Основной поток

1. Руководитель сервиса открывает карточку заявки → нажимает **«Отменить заявку»**.
2. Система проверяет допустимость перехода (см. таблицу выше).
3. Система отображает диалог подтверждения с полем **«Причина отмены»** (обязательно).
4. При статусе «Назначена» или «В работе» — дополнительное предупреждение: *«На заявку назначен инженер [ФИО]. Он будет уведомлён об отмене»*.
5. Руководитель сервиса вводит причину и нажимает **«Подтвердить отмену»**.
6. Система:
   - Переводит заявку в статус **«Отменена»**
   - Останавливает SLA-таймер (`sla_stopped_at = NOW()`)
   - Записывает причину отмены (`cancellation_reason`)
   - Если был назначен инженер — отправляет уведомление об отмене
   - Если оборудование было в статусе «На ремонте» — возвращает статус оборудования в **«Активно»** (автоматически, как при закрытии заявки)
   - Фиксирует смену статуса в аудит-логе
7. Система отображает сообщение: *«Заявка [N] отменена»*.

## Альтернативные потоки

**АП-1: Отмена с предложением создания новой заявки-дубликата**
Если причина отмены — ошибка при создании (дубликат), система предлагает: *«Хотите создать заявку с теми же данными?»*. При согласии — открывает форму создания с предзаполненными полями.

---

## Исключения

**ИП-1: Попытка отменить заявку в статусе «Выполнена», «На проверке» или «Закрыта»**
→ Кнопка «Отменить заявку» недоступна. Тултип: *«Нельзя отменить заявку с текущим статусом. Для возврата в работу — используйте смену статуса (→ UC-903)»*.

**ИП-2: Причина отмены не указана**
→ Кнопка «Подтвердить отмену» неактивна до заполнения поля.

**ИП-3: Заявка уже отменена**
→ Кнопка «Отменить заявку» не отображается.

---

## Ключевые данные

| Поле | Тип | Описание |
|---|---|---|
| cancellation_reason | text | Причина отмены (обязательна, хранится в tickets) |
| cancelled_by | FK → users | Кто отменил |
| cancelled_at | timestamp | Дата и время отмены |
| sla_stopped_at | timestamp | Момент остановки SLA-таймера |

> Поля `cancellation_reason`, `cancelled_by`, `cancelled_at` требуют добавления в таблицу `tickets` (текущая схема их не содержит).

## RBAC

| Действие | Роли |
|---|---|
| Отменить заявку | `admin`, `svc_mgr` |
| Просмотр отменённых заявок | все авторизованные |

Инженер отменить заявку не может — только Руководитель сервиса и Администратор.

  Scenario: Успешная отмена новой заявки
    Given заявка #SR-100 в статусе «Новая»

    Given заявка #SR-102 на оборудование SN-555 в статусе «В работе»

    When заявка отменяется
    Then статус SN-555 = «Активно»

  Scenario: Нельзя отменить завершённую заявку
    Given заявка #SR-103 в статусе «Выполнена»

    Given заявка #SR-104 в статусе «Новая»
    When пользователь пытается подтвердить отмену без указания причины
    Then кнопка «Подтвердить отмену» неактивна

## Примечания

- Отменённые заявки хранятся в системе (no soft delete на уровне записи) и видны в списке заявок с фильтром «Отменена». История изменений статусов доступна.
- Отменённая заявка не учитывается в статистике SLA как «нарушение».
- Обновление UC-903: в диаграмму жизненного цикла добавить переход из статусов `new`, `assigned`, `in_progress`, `waiting_part` в `cancelled`.
- В ER: добавить поля `cancellation_reason TEXT`, `cancelled_by FK → users`, `cancelled_at TIMESTAMP` в таблицу `tickets`.
