# UC-801: Создать / редактировать карточку сотрудника

**Версия:** 1.0 | **Дата:** 25.03.2026 | **Статус:** Согласован

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-801 |
| **Модуль** | Модуль 8: Учёт персонала |
| **Основной актор** | Администратор |
| **Второстепенные акторы** | Система (soft delete, аудит-лог) |
| **Связанные BR-F** | BR-F-800, BR-F-801 |
| **Связанные BR-R** | BR-R-009 (soft delete), BR-R-010 (аудит-лог) |
| **Приоритет** | Must |
| **Предусловия** | Пользователь авторизован с ролью **Администратор** |
| **Постусловия** | Карточка сотрудника создана / обновлена и сохранена в системе; изменение зафиксировано в аудит-логе |
| **Триггер** | Администратор открывает форму создания нового сотрудника или выбирает «Редактировать» на карточке существующего |

---

## Основной поток — Создание

1. Администратор переходит в раздел **«Сотрудники»** → нажимает **«+ Добавить сотрудника»**.
2. Система отображает пустую форму карточки сотрудника.
3. Администратор заполняет обязательные поля: ФИО, должность, подразделение, рабочий телефон, email.
4. Администратор выбирает статус: **Активен** (по умолчанию).
5. Администратор нажимает **«Сохранить»**.
6. Система валидирует данные (уникальность email, формат телефона).
7. Система создаёт карточку, присваивает ID, фиксирует запись в аудит-логе (`action: create`, `entity: user`, `user_id: ID сотрудника`).
8. Система отображает созданную карточку с сообщением **«Сотрудник добавлен»**.

---

## Основной поток — Редактирование

1. Администратор открывает карточку сотрудника → нажимает **«Редактировать»**.
2. Система переводит форму в режим редактирования.
3. Администратор изменяет нужные поля.
4. Администратор нажимает **«Сохранить»**.
5. Система валидирует данные.
6. Система сохраняет изменения, фиксирует в аудит-логе (`action: update`, старое и новое значение изменённого поля).
7. Система отображает сообщение **«Изменения сохранены»**.

---

## Альтернативные потоки

**АП-1: Изменение статуса на «В отпуске»**
На шаге 3 (Редактирование): Администратор меняет статус на **«В отпуске»** и указывает дату возвращения (опционально).
Система сохраняет. При назначении заявок на этого инженера будет показываться предупреждение (→ UC-901, шаг назначения).

**АП-2: Деактивация сотрудника (увольнение)**
На шаге 3 (Редактирование): Администратор меняет статус на **«Уволен»**.
Система применяет **soft delete** (BR-R-009): `is_deleted = true`, карточка исчезает из рабочих списков, но остаётся в аудит-логе и истории заявок.
Если у сотрудника есть открытые заявки — система показывает предупреждение: *«У сотрудника есть N открытых заявок. Переназначьте их перед деактивацией»*. Переход к деактивации блокируется до снятия заявок.

---

## Исключения

**ИП-1: Email уже занят**
На шаге 6: Система обнаруживает, что введённый email уже есть у другого сотрудника.
→ Форма показывает ошибку рядом с полем: *«Этот email уже используется: [Имя сотрудника]»*. Сохранение блокируется.

**ИП-2: Обязательные поля не заполнены**
На шаге 6: Незаполненные обязательные поля подсвечиваются красным.
→ Сообщение: *«Заполните обязательные поля»*. Сохранение блокируется.

## Ключевые данные

| Поле | Обязательное | Тип | Ограничения |
|---|---|---|---|
| ФИО | Да | varchar(200) | Не пустое |
| Должность | Да | varchar(100) | Не пустое |
| Подразделение | Да | varchar(100) | Не пустое |
| Рабочий телефон | Да | varchar(20) | Формат: +XXXXXXXXXXX |
| Email | Да | varchar(150) | Уникальный, формат email |
| Дата приёма | Нет | date | Не позже текущей даты |
| Статус | Да | enum | Активен / В отпуске / Уволен (по умолчанию: Активен) |

## Acceptance Criteria

```gherkin
Feature: Управление карточкой сотрудника

  Scenario: Успешное создание сотрудника
    Given пользователь авторизован как Администратор
    When он заполняет форму с валидными данными (ФИО, должность, подразделение, телефон, email)
    And нажимает «Сохранить»
    Then система создаёт карточку сотрудника
    And отображает сообщение «Сотрудник добавлен»
    And в аудит-логе появляется запись action=create, entity=user

  Scenario: Блокировка при дублировании email
    Given в системе уже есть сотрудник с email "ivan@company.com"
    When Администратор пытается создать нового сотрудника с email "ivan@company.com"
    Then система показывает ошибку «Этот email уже используется»
    And сохранение блокируется

  Scenario: Деактивация сотрудника без открытых заявок
    Given сотрудник не имеет открытых заявок
    When Администратор меняет статус на «Уволен» и сохраняет
    Then is_deleted = true для данного сотрудника
    And карточка исчезает из рабочих списков
    And в аудит-логе фиксируется action=update, field=status, old=Активен, new=Уволен

  Scenario: Блокировка деактивации при наличии открытых заявок
    Given сотрудник имеет 2 открытые заявки
    When Администратор меняет статус на «Уволен» и нажимает «Сохранить»
    Then система показывает предупреждение «У сотрудника есть 2 открытых заявки. Переназначьте их перед деактивацией»
    And деактивация не выполняется
```

## Примечания / Бизнес-правила

- **BR-R-009 (Soft delete):** физическое удаление записей запрещено во всех модулях. Уволенный сотрудник помечается `is_deleted=true`, не отображается в интерфейсе, но доступен в аудит-логе.
- **BR-R-010 (Аудит-лог):** любое изменение статуса и данных сотрудника автоматически фиксируется с датой, пользователем и старым/новым значением.
- Назначение роли — отдельный UC-802. Карточка сотрудника и учётная запись с ролью создаются в разных потоках.
