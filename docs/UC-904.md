## Основной поток

1. Инженер открывает заявку (статус «В работе»), переходит на вкладку **«Акт выполненных работ»**.
2. Система отображает форму акта (пустую или частично заполненную).
3. Система устанавливает **дату работ** (`work_date`) = текущая дата. Разрешено изменить на прошедшую дату (постфактум). Будущая дата недопустима.
4. Инженер заполняет **описание выполненных работ** (свободный текст).
5. Инженер вносит **время в пути** и **время работы** (BR-F-805).
6. Инженер добавляет **позиции акта** — услуги и запчасти:

   а. Нажимает «Добавить позицию».

   б. Выбирает тип: **Услуга** (из ServiceCatalog) или **Запчасть** (из SparePart).

   в. Система подставляет наименование, ед. измерения и цену из справочника (BR-F-111).

   г. **Для запчастей:** Система определяет и подставляет склад-источник по умолчанию (приоритет: склад банка клиента → при наличии остатка; иначе — основной склад организации).

   д. **Для запчастей:** Система отображает **доступный остаток** по выбранному складу.

   е. **Для запчастей со склада банка (`warehouse.type = bank`):** Система автоматически устанавливает `unit_price = 0` и блокирует поле цены (BR-P-010).

   ж. Инженер при необходимости меняет склад-источник или количество.

   з. Инженер вправе изменить цену для запчастей с основного склада.

7. Инженер прикрепляет **фото**: минимум 1, максимум 10, JPEG/PNG, до 10 МБ.
8. Инженер фиксирует **подтверждение клиента**: ФИО представителя + способ (`on_device` / `email_link` / `verbal`).
9. Инженер нажимает **«Сохранить акт»**.
10. Система выполняет валидацию (см. раздел «Проверки»).
11. Система проводит акт в рамках одной транзакции (см. «Алгоритм проведения»).
12. Система обновляет историю ремонтов оборудования (BR-F-906).
13. Система отображает: *«Акт сохранён. Можно переводить заявку в статус "Выполнена"»*.

## Альтернативные потоки

**АП-0: Редактирование существующего несохранённого акта (BR-F-114)**
1. Инженер открывает заявку с уже сохранённым неподписанным актом.
2. Система отображает кнопку **«Редактировать акт»** (только если `signed_by = null`).
3. Система открывает форму с предзаполненными данными, включая склады-источники по позициям.
4. Инженер вносит изменения (шаги 4–8 основного потока).
5. При сохранении: отменяются предыдущие списания → проводятся новые по актуальным позициям.
6. Система отображает: *«Акт обновлён»*.

**АП-1: Приоритет склада банка клиента**
Система проверяет: существует ли склад `type=bank` для `client_id` оборудования из заявки, и есть ли на нём остаток запчасти.
- Если да → устанавливает этот склад как источник по умолчанию, цена = 0.
- Если нет (склад отсутствует или qty = 0) → устанавливает основной склад `type=company`.
Инженер может вручную сменить склад-источник.

**АП-2: Инженер выбирает другой склад**
На шаге 6ж инженер меняет склад-источник в позиции:
- Система пересчитывает `available_qty` по новому складу.
- Если новый склад `type=bank` → `unit_price = 0`, поле заблокировано.
- Если новый склад `type=company` → `unit_price` восстанавливается из справочника SparePart, поле доступно.

**АП-3: Нехватка запчасти на выбранном складе**
При сохранении система обнаруживает, что `quantity > WarehouseStock(warehouse_id, part_id).quantity`.
→ Система блокирует сохранение и показывает: *«На складе [название] доступно [N] шт. [Артикул], запрошено [M]. Скорректируйте количество или выберите другой склад»*.

**АП-4: Гарантийный ремонт — данные для рекламации**
Если `is_warranty = true`, система дополнительно сохраняет данные акта в структуру рекламации вендору (→ BR-F-1111, Модуль 11).

**АП-5: Подписание акта представителем клиента (BR-F-116, BR-F-117)**
1. Представитель клиента (роль `client_user`) открывает карточку заявки.
2. Система отображает акт со статусом **«Не подписан»** и кнопкой **«Подписать акт»**.
3. Представитель нажимает «Подписать акт».
4. Система фиксирует: `signed_by = user_id`, `signed_at = now()`.
5. Статус акта: **«Подписан»** с датой и ФИО.

**АП-6: Отображение связанного счёта (BR-F-119, BR-F-120)**
После создания счёта из акта — рядом с актом отображается метка «Счёт #NNN» со ссылкой и статусом оплаты.

## Проверки при сохранении

| № | Проверка | Ошибка |
|---|---|---|
| 1 | `work_description` не пустое | «Заполните описание выполненных работ» |
| 2 | `photos.length ≥ 1` | «Прикрепите минимум 1 фотографию» |
| 3 | `work_date ≤ сегодня` | «Дата работ не может быть позже текущей даты» |
| 4 | `client_confirmed_by_name` заполнено | «Укажите ФИО представителя клиента» |
| 5 | `client_confirmed_method` выбран | «Укажите способ подтверждения выполненных работ» |
| 6 | Для каждой части: `quantity ≥ 1` | «Количество должно быть не менее 1» |
| **7** | **Для каждой части: `quantity ≤ WarehouseStock(warehouse_id, part_id).quantity`** | **«На складе [название] доступно [N] шт. [name], запрошено [M]»** |
| **8** | **`warehouse_id` ссылается на активный склад** | **«Склад недоступен»** |
| 9 | Файл фото: JPEG/PNG, ≤ 10 МБ | «Допустимые форматы: JPEG, PNG. Максимальный размер: 10 МБ» |

## Алгоритм проведения (транзакция)

При сохранении акта в рамках одной DB-транзакции:

1. Заблокировать строки `WarehouseStock` для всех `(warehouse_id, part_id)` из позиций типа `part` (`SELECT ... FOR UPDATE`).
2. Проверить остатки по каждой позиции (ошибка → откат).
3. Для каждой позиции типа `part`:
   - Уменьшить `WarehouseStock(item.warehouse_id, item.part_id).quantity` на `item.quantity`.
   - Если `quantity` стало отрицательным → откат с ошибкой.
4. Сохранить акт (`WorkAct`) и все позиции (`WorkActItem`) с зафиксированными значениями (`unit_price_snapshot = item.unit_price`).
5. Обновить историю ремонтов оборудования.
6. Зафиксировать в аудит-логе: `action=WORK_ACT`, список позиций-запчастей с qty и warehouse.

> **При редактировании акта (АП-0):** перед повторным проведением выполнить обратную операцию (вернуть qty) по предыдущим позициям, затем провести новые.

## Счёт из акта — ценообразование

При создании счёта из акта (`POST /invoices/from-act/{act_id}`):

| Условие строки | Цена в счёте |
|---|---|
| `item_type = service` | `unit_price` из акта |
| `item_type = part`, `warehouse.type = company` | `unit_price` из акта |
| `item_type = part`, `warehouse.type = bank` | **0** (BR-P-010) |

> Цена фиксируется на момент создания счёта и не изменяется при изменении прайс-листа (BR-P-003).

## Бизнес-правила

- **BR-R-004:** кнопка «Закрыта» активируется только после сохранения акта.
- **BR-R-005:** списание запчастей без привязки к заявке запрещено. Акт = основание для списания.
- **BR-R-006:** для гарантийных заявок данные акта — основа для рекламации вендору (Модуль 11).
- **BR-P-010:** если запчасть берётся со склада банка (`warehouse.type = bank`) — `unit_price` в акте и счёте = 0.
- **BR-P-001:** остатки хранятся в разрезе `(warehouse_id, part_id)`; списание уменьшает `WarehouseStock` по указанному складу.
- **BR-P-003:** цена в акте и счёте фиксируется на момент создания и не меняется при изменении прайс-листа.
- **BR-F-908:** сохранение акта возможно только при наличии клиентского подтверждения.
- **BR-F-116:** подписание акта — исключительно роль `client_user`.
- **BR-F-117:** статус подписи всегда виден в карточке заявки.
- **BR-F-118:** кнопка «Создать счёт из акта» активна только если в акте есть хотя бы одна позиция.

## UI: изменения формы акта (v1.4)

В табличной части «Позиции акта» для строк типа **Запчасть** добавляется:

## Примечания

- `work_date` по умолчанию = текущая дата. Разрешена корректировка на прошедшую дату (постфактум). Будущая дата запрещена.
- Акт заполняется из мобильного приложения (на выезде) и через веб-интерфейс.
- При редактировании акта система выполняет обратное списание предыдущих позиций перед повторным проведением.
- `unit_price_snapshot` в `WorkActItem` фиксирует цену на момент сохранения — изменение прайс-листа не влияет на уже сохранённые акты.
