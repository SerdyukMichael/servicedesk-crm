# UC-904: Заполнить акт выполненных работ

**Версия:** 1.3 | **Дата:** 08.04.2026 | **Статус:** В работе

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-904 |
| **Модуль** | Модуль 9: Заявки на обслуживание |
| **Основной актор** | Сервисный инженер |
| **Второстепенные акторы** | Контактное лицо клиента (подтверждение работ), Система (списание запчастей → BR-F-907, обновление истории ремонтов → BR-F-906) |

## Основной поток

1. Инженер открывает заявку (статус «В работе»), переходит на вкладку **«Акт выполненных работ»**.
2. Система отображает форму акта (пустую или частично заполненную, если работа прерывалась).
2а. Система устанавливает **дату работ** (`work_date`) = текущая дата по умолчанию. Инженер может изменить на более раннюю дату (для заполнения акта постфактум). Будущая дата недопустима.
3. Инженер заполняет **описание выполненных работ** (свободный текст).
4. Инженер вносит **время в пути**: часы и минуты (BR-F-805).
5. Инженер вносит **время работы**: часы и минуты (BR-F-805).
6. Инженер добавляет **позиции акта** — услуги и запчасти:

   а. Нажимает «Добавить позицию».

   б. Выбирает тип: **Услуга** (из справочника ServiceCatalog) или **Запчасть** (из SparePart).

   в. Система автоматически подставляет наименование, единицу и цену из прайса (BR-F-111).

   г. Инженер указывает количество. Система вычисляет сумму строки (`qty × unit_price`).

   д. Инженер вправе изменить цену (например, гарантийный ремонт → 0 руб.).

   е. Для запчастей: инженер выбирает источник списания — основной склад или передвижной склад.
7. Инженер прикрепляет **фото**: минимум 1 фото (до ремонта или после), максимум 10, формат JPEG/PNG, до 10 МБ каждое.
7а. Инженер фиксирует **подтверждение клиента**: вводит ФИО представителя клиента и выбирает способ подтверждения:
   - **На устройстве** — представитель клиента ставит подпись / нажимает «Подтверждаю» на экране инженера;
   - **Ссылка на email** — система отправляет ссылку для подтверждения на email клиентской организации;
   - **Устно + ФИО** — инженер вносит ФИО представителя, подтвердившего устно.
8. Инженер нажимает **«Сохранить акт»**.
9. Система валидирует данные (наличие описания, минимум 1 фото, наличие запчастей на складе, заполнено ФИО представителя клиента и способ подтверждения).
10. Система сохраняет акт.
11. Система автоматически списывает указанные запчасти со склада (BR-F-907), создавая записи расхода с привязкой к номеру заявки.
12. Система обновляет запись в истории ремонтов оборудования (BR-F-906): добавляет использованные запчасти и описание работ.
13. Система отображает: *«Акт сохранён. Можно переводить заявку в статус "Выполнена"»*.

## Альтернативные потоки

**АП-0: Редактирование существующего несписанного акта (BR-F-114)**
1. Инженер открывает заявку, в которой уже есть сохранённый акт (не подписанный).
2. Система отображает кнопку **«Редактировать акт»** рядом с просмотром акта.
3. Система проверяет: `signed_by = null`. Если проверка пройдена — открывает форму, предзаполненную данными текущего акта (описание, время, позиции).
4. Инженер вносит изменения (шаги 3–6 основного потока).
5. Инженер нажимает **«Сохранить акт»**.
6. Система сохраняет изменения: обновляет поля акта и **полностью заменяет список позиций** (удаляет старые, вставляет новые).
7. Система отображает: *«Акт обновлён»*.

**АП-1: Запчасти на передвижном складе инженера**
На шаге 6 инженер переключает источник списания на «Мой склад».
Система отображает остатки передвижного склада инженера.
При сохранении — списание происходит с передвижного склада.

**АП-2: Нехватка запчасти на складе**
При сохранении система обнаруживает, что запрошенное количество превышает остаток.
→ Система предупреждает: *«На складе [X] шт. [Артикул], запрошено [Y]. Скорректируйте количество или замените источник»*.
→ Сохранение блокируется для конкретной позиции.

**АП-3: Гарантийный ремонт — формирование данных для рекламации**
Если `is_warranty = true`, система дополнительно сохраняет данные акта в структуру рекламации вендору (→ BR-F-1111, Модуль 11) для последующего оформления.

**АП-4: Подписание акта представителем клиента (BR-F-116, BR-F-117)**
1. Представитель клиента (роль `client_user`) открывает карточку заявки.
2. Система отображает акт со статусом **«Не подписан»** и кнопкой **«Подписать акт»**.
3. Представитель нажимает «Подписать акт».
4. Система фиксирует: `signed_by = user_id`, `signed_at = now()`.
5. Статус акта меняется на **«Подписан»** с указанием даты и ФИО.
6. Кнопки «Редактировать акт» и «Подписать акт» скрываются.
7. Роли `engineer`, `svc_mgr`, `admin` не видят кнопку «Подписать акт» (BR-F-116).

**АП-5: Отображение связанного счёта и статуса оплаты (BR-F-119, BR-F-120)**
1. После создания счёта из акта система отображает рядом с актом метку **«Счёт #NNN»** со ссылкой.
2. Если счёт оплачен (`is_paid = true`) — рядом отображается бейдж **«Оплачен»**.
3. Если счёт не оплачен — бейдж **«Не оплачен»**.
4. Статус обновляется при изменении поля `is_paid` на счёте.

## Исключения

**ИП-1: Не прикреплено ни одного фото**
→ Система блокирует: *«Прикрепите минимум 1 фотографию (до или после ремонта)»*.

**ИП-2: Описание работ не заполнено**
→ Система блокирует: *«Заполните описание выполненных работ»*.

**ИП-3: Файл не соответствует требованиям**
→ Система показывает ошибку рядом с файлом: *«Допустимые форматы: JPEG, PNG. Максимальный размер: 10 МБ»*.

**ИП-4: Не заполнено подтверждение клиента**
→ Система блокирует: *«Укажите ФИО представителя клиента и способ подтверждения выполненных работ»*.

**ИП-5: Дата работ в будущем**
→ Система блокирует: *«Дата работ не может быть позже текущей даты»*.

**ИП-6: Попытка редактирования подписанного акта (BR-F-115)**
→ Система возвращает 403 Forbidden: *«Акт подписан и не может быть изменён»*. Кнопка «Редактировать акт» скрыта, если `signed_by` заполнен.

**ИП-7: Попытка создать счёт при отсутствии позиций в акте (BR-F-118)**
→ Кнопка «Создать счёт из акта» недоступна (disabled) если `items = []`. При наведении — подсказка: *«Добавьте позиции в акт перед созданием счёта»*.

**ИП-8: Попытка подписать акт пользователем без роли `client_user` (BR-F-116)**
→ Кнопка «Подписать акт» не отображается для ролей `engineer`, `svc_mgr`, `admin`. При прямом API-вызове — 403 Forbidden.

## Ключевые данные

| Поле | Обязательное | Тип | Ограничения |
|---|---|---|---|
| work_description | Да | text | Не пустое |
| work_date | Да | date | По умолчанию = текущая дата; допустимо только прошлое или сегодня |
| travel_time_minutes | Нет | integer | ≥ 0 |
| work_time_minutes | Нет | integer | ≥ 0 |
| items | Нет | []WorkActItem | Каждая позиция: тип (service/part), ссылка на справочник, наименование, кол-во, ед., цена, сумма |
| photos | Да | []file | Минимум 1, максимум 10, JPEG/PNG, ≤ 10 МБ |
| completed_at | Да | timestamp | Авто: момент сохранения акта |
| client_confirmed | Да | boolean | true = клиент подтвердил выполнение |
| client_confirmed_by_name | Да | varchar(255) | ФИО представителя клиента |
| client_confirmed_at | Да | timestamp | Авто: момент фиксации подтверждения |
| client_confirmed_method | Да | enum | `on_device`, `email_link`, `verbal` |

## Acceptance Criteria

```gherkin
Feature: Акт выполненных работ

  Scenario: Успешное сохранение акта с запчастями
    Given заявка #42 в статусе «В работе», на складе инженера есть «Картридж XID» 5 шт.
    When инженер заполняет: описание работ, 30 мин. в пути, 2 ч. работы, 1 картридж, 1 фото
    And нажимает «Сохранить акт»
    Then акт сохранён
    And остаток «Картридж XID» на складе = 4 шт.
    And в истории ремонтов оборудования по заявке добавлена запись с деталями
    And статус заявки можно изменить на «Выполнена»

  Scenario: Блокировка при отсутствии фото
    When инженер заполняет акт без прикреплённых фотографий
    Then система показывает ошибку «Прикрепите минимум 1 фотографию»
    And сохранение блокируется

  Scenario: Предупреждение при нехватке запчасти
    Given на складе 2 шт. «Ролик подачи»
    When инженер указывает 3 шт. «Ролик подачи»
    Then система показывает «На складе 2 шт., запрошено 3»
    And сохранение блокируется до корректировки

  Scenario: Успешное сохранение акта с подтверждением клиента
    Given заявка #42 в статусе «В работе»
    When инженер заполняет акт, вводит ФИО «Иванов А.А.» и выбирает способ «На устройстве»
    And нажимает «Сохранить акт»
    Then акт сохранён с client_confirmed = true, client_confirmed_by_name = «Иванов А.А.», method = on_device
    And статус заявки можно изменить на «Выполнена»

  Scenario: Блокировка при отсутствии подтверждения клиента
    Given инженер заполнил описание и прикрепил фото, но не указал ФИО представителя клиента
    When инженер нажимает «Сохранить акт»
    Then система показывает ошибку «Укажите ФИО представителя клиента и способ подтверждения»
    And сохранение блокируется

  Scenario: Дата работ изменена на прошедшую (постфактум)
    Given инженер открывает акт по заявке #55
    When он меняет дату работ с 27.03.2026 на 25.03.2026
    And заполняет акт полностью и нажимает «Сохранить акт»
    Then акт сохранён с work_date = 25.03.2026

  Scenario: Блокировка при будущей дате работ
    Given инженер вводит дату работ 01.04.2026 (будущая дата)
    When нажимает «Сохранить акт»
    Then система показывает ошибку «Дата работ не может быть позже текущей даты»
    And сохранение блокируется

  Scenario: Закрытие заявки невозможно без сохранённого акта
    Given заявка в статусе «На проверке», акт не заполнен
    Then кнопка «Закрыта» не активна (BR-R-004)

  Scenario: Подписание акта только представителем клиента
    Given акт сохранён, пользователь — client_user клиентской организации
    When представитель клиента нажимает «Подписать акт»
    Then signed_by = id пользователя, signed_at = текущее время
    And статус акта отображается как «Подписан»
    And кнопка «Подписать акт» скрывается

  Scenario: Инженер не видит кнопку «Подписать акт»
    Given акт сохранён, пользователь — engineer
    Then кнопка «Подписать акт» не отображается

  Scenario: Статус акта «Не подписан» виден всем
    Given акт сохранён и не подписан
    Then в карточке заявки отображается статус «Не подписан»

  Scenario: Блокировка создания счёта без позиций
    Given акт сохранён без позиций (items = [])
    Then кнопка «Создать счёт из акта» недоступна
    And при наведении — подсказка «Добавьте позиции в акт»

  Scenario: Отображение созданного счёта рядом с актом
    Given по акту создан счёт #42
    Then рядом с актом отображается метка «Счёт #42» со ссылкой

  Scenario: Статус оплаты счёта рядом с актом
    Given по акту создан счёт #42, счёт оплачен
    Then рядом с актом отображается бейдж «Оплачен»

  Scenario: Неоплаченный счёт
    Given по акту создан счёт #42, счёт не оплачен
    Then рядом с актом отображается бейдж «Не оплачен»
```

## Примечания / Бизнес-правила

- **BR-R-004:** кнопка «Закрыта» активируется только после сохранения акта.
- **BR-R-005:** списание запчастей без привязки к заявке запрещено. Акт = основание для списания.
- **BR-R-006:** для гарантийных заявок данные акта служат основой для рекламации вендору (Модуль 11).
- **BR-F-908:** сохранение акта возможно только при наличии клиентского подтверждения. ФИО представителя и способ подтверждения — обязательные поля.
- **BR-F-116:** подписание акта — исключительно роль `client_user`. Остальные роли подписать акт не могут.
- **BR-F-117:** статус подписи («Не подписан» / «Подписан [дата, ФИО]») всегда виден в карточке заявки.
- **BR-F-118:** кнопка «Создать счёт из акта» активна только если в акте есть хотя бы одна позиция.
- **BR-F-119:** ссылка на созданный счёт отображается рядом с актом — пользователь сразу видит связь.
- **BR-F-120:** статус оплаты счёта («Оплачен» / «Не оплачен») отображается рядом с актом; обновляется при изменении счёта.
- `work_date` по умолчанию = текущая дата. Разрешена корректировка на прошедшую дату для постфактум-заполнения. Будущая дата запрещена.
- Акт заполняется из мобильного приложения (на выезде). Веб-интерфейс также доступен.
- Акт можно сохранять частично (черновик) и дополнять — до окончательного сохранения. Подтверждение клиента фиксируется при финальном сохранении.
