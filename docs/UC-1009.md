# UC-1009: Управлять справочником моделей оборудования

**Версия:** 1.0 | **Дата:** 31.03.2026 | **Статус:** Рабочий

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-1009 |
| **Модуль** | Модуль 10: Учёт оборудования |
| **Основной актор** | Руководитель сервиса |
| **Второстепенные акторы** | Администратор (те же права), Система (аудит-лог) |
| **Связанные BR-F** | BR-F-1000, BR-F-1003 |
| **Связанные BR-R** | BR-R-009 (soft delete), BR-R-010 (аудит-лог) |
| **Приоритет** | Must |
| **Предусловия** | Пользователь авторизован с ролью **Руководитель сервиса** или **Администратор** |
| **Постусловия** | Запись справочника создана / обновлена / деактивирована; изменения применяются при следующем обращении к справочнику |
| **Триггер** | Пользователь открывает **«Настройки» → «Справочник моделей оборудования»** |

> **Критичность:** справочник является предусловием для UC-803 (компетенции инженеров), UC-901 (создание заявки), UC-1001 (паспорт оборудования), UC-908 (график ТО). Без этого UC справочник не имеет управляемого жизненного цикла.

---

## Основной поток — Создание модели

1. Пользователь переходит в **«Настройки» → «Справочник моделей»** → нажимает **«+ Добавить модель»**.
2. Система отображает форму создания.
3. Пользователь заполняет обязательные поля: название, категория.
4. Опционально: производитель, описание, срок гарантии по умолчанию (в месяцах).
5. Пользователь нажимает **«Сохранить»**.
6. Система валидирует уникальность названия в рамках производителя.
7. Система создаёт запись с `is_active = true`, фиксирует в аудит-логе (`action: create`, `entity: equipment_model`).
8. Система отображает сообщение **«Модель добавлена»**.

---

## Основной поток — Редактирование модели

1. Пользователь открывает строку модели → нажимает **«Редактировать»**.
2. Система отображает форму с заполненными значениями.
3. Пользователь изменяет нужные поля.
4. Нажимает **«Сохранить»**.
5. Система валидирует и сохраняет, фиксирует в аудит-логе (`action: update`).
6. Система отображает **«Изменения сохранены»**.

> Изменение названия или категории автоматически отражается во всех паспортах оборудования и карточках компетенций, где используется эта модель.

---

## Основной поток — Деактивация модели

1. Пользователь нажимает **«Деактивировать»** у нужной строки.
2. Система показывает подтверждение: *«Деактивировать модель [Название]? Она перестанет отображаться при создании нового оборудования и назначении заявок. Существующие паспорта и компетенции не затрагиваются»*.
3. Пользователь подтверждает.
4. Система устанавливает `is_active = false`, фиксирует в аудит-логе.
5. Модель скрывается из выпадающих списков при создании нового оборудования; остаётся видимой в уже существующих паспортах.

---

## Альтернативные потоки

**АП-1: Восстановление деактивированной модели**
Пользователь переключает список на «Показать деактивированные» → нажимает **«Активировать»**.
Система устанавливает `is_active = true`, модель снова доступна в выпадающих списках.

**АП-2: Просмотр списка с фильтрами**
Доступные фильтры: категория оборудования, производитель, статус (активные / деактивированные / все).

---

## Исключения

**ИП-1: Дублирование названия у одного производителя**
→ Система блокирует: *«Модель с таким названием у производителя [Название] уже существует»*.

**ИП-2: Попытка деактивировать модель с активными паспортами**
→ Система показывает предупреждение: *«У модели [Название] есть N единиц оборудования в статусе «Активно». После деактивации новые паспорта на эту модель создать нельзя, но существующие остаются»*. Деактивация не блокируется — только информирование.

## Ключевые данные

| Поле | Обязательное | Тип | Описание |
|---|---|---|---|
| name | Да | varchar(255) | Наименование модели |
| manufacturer | Нет | varchar(128) | Производитель |
| category | Да | enum | `atm` / `card_printer` / `pos_terminal` / `other` |
| description | Нет | text | Описание |
| warranty_months_default | Нет | integer | Срок гарантии по умолчанию (месяцев) |
| is_active | Да | boolean | true — активна (по умолчанию), false — деактивирована |

## Acceptance Criteria

```gherkin
Feature: Справочник моделей оборудования

  Scenario: Успешное создание модели
    Given пользователь авторизован как Руководитель сервиса
    When он добавляет модель «Matica XID 580i», категория «Принтер карт»
    Then модель появляется в справочнике со статусом «Активна»
    And доступна для выбора при создании нового оборудования (UC-1001)
    And доступна для добавления в профиль компетенций инженера (UC-803)

  Scenario: Блокировка при дублировании названия
    Given модель «Matica XID 580i» уже существует
    When пользователь пытается добавить «Matica XID 580i» того же производителя
    Then система показывает ошибку «Модель с таким названием уже существует»

  Scenario: Деактивация модели с предупреждением
    Given модель «Matica XID 580i» используется в 5 активных паспортах
    When пользователь деактивирует модель
    Then система показывает предупреждение о 5 паспортах
    And после подтверждения is_active = false
    And модель не отображается в списке при создании нового паспорта
    And существующие 5 паспортов по-прежнему ссылаются на эту модель

  Scenario: Восстановление деактивированной модели
    Given модель «Matica XID 580i» деактивирована
    When пользователь активирует её из раздела «Деактивированные»
    Then is_active = true
    And модель снова доступна для выбора при создании паспорта
```

---

## Примечания

- Справочник является глобальным — общий для всех клиентов и модулей.
- UC-803 содержит условие «справочник моделей оборудования заполнен» — данный UC обеспечивает это предусловие.
- При необходимости добавить поле `warranty_months_default` в таблицу `equipment_models` (сейчас отсутствует в ER) — создать Alembic-миграцию.
