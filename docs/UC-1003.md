# UC-1003: Управлять статусом и передачей оборудования

**Версия:** 1.0 | **Дата:** 26.03.2026 | **Статус:** На проверке

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-1003 |
| **Модуль** | Модуль 10: Учёт оборудования |
| **Основной актор** | Руководитель сервиса |
| **Второстепенные акторы** | Система (авто-статус «На ремонте» при открытой заявке) |
| **Связанные BR-F** | BR-F-1007, BR-F-1010 |
| **Связанные BR-R** | BR-R-009 (история передач не удаляется) |
| **Приоритет** | Must |
| **Предусловия** | Паспорт оборудования существует; пользователь авторизован с ролью **Руководитель сервиса** |
| **Постусловия** | Статус оборудования изменён; история изменений сохранена |
| **Триггер** | Пользователь открывает паспорт оборудования и изменяет статус |

---

## Статусы оборудования и переходы

| Статус | Кто/что устанавливает | Условие |
|---|---|---|
| Активно | Ручной / Авто (при создании и закрытии заявки) | — |
| На ремонте | Авто: при переходе заявки в «В работе» | Открытая заявка на это оборудование |
| Списано | Руководитель сервиса (ручной) | Необратимый статус |
| Передано другому клиенту | Руководитель сервиса (ручной, через форму передачи) | — |

---

## Основной поток — Списание оборудования

1. Пользователь открывает паспорт → нажимает **«Списать оборудование»**.
2. Система запрашивает подтверждение: *«Списать оборудование [Модель] SN-[X]? Это действие необратимо»*.
3. Пользователь вводит причину списания (обязательно).
4. Подтверждает.
5. Система устанавливает статус = **«Списано»**, фиксирует дату и причину, записывает в аудит-лог.

---

## Основной поток — Передача другому клиенту

1. Пользователь открывает паспорт → нажимает **«Передать клиенту»**.
2. Система открывает форму: новый клиент (выбор из справочника), дата передачи.
3. Пользователь заполняет и подтверждает.
4. Система:
   - Меняет `client_id` на нового клиента
   - Добавляет запись в историю передач: дата, от кого, кому
   - Обнуляет адрес установки (требует повторного заполнения)
   - Фиксирует в аудит-логе

---

## Исключения

**ИП-1: Списание оборудования с открытой заявкой**
→ Система блокирует: *«Нельзя списать оборудование с открытой заявкой #[N]. Закройте заявку сначала»*.

**ИП-2: Передача оборудования с открытой заявкой**
→ Система предупреждает (не блокирует): *«У оборудования есть открытая заявка #[N]. Уверены?»*

## Acceptance Criteria

```gherkin
Feature: Управление статусом оборудования

  Scenario: Авто-переход в статус «На ремонте»
    Given заявка #42 на оборудование SN-12345 переводится в «В работе»
    Then статус SN-12345 = «На ремонте» автоматически

  Scenario: Возврат статуса «Активно» при закрытии заявки
    Given заявка #42 закрывается
    Then статус SN-12345 = «Активно»

  Scenario: Списание оборудования
    Given оборудование SN-99999 без открытых заявок
    When Руководитель сервиса списывает с причиной «Физический износ»
    Then статус = «Списано», дата и причина зафиксированы

  Scenario: Передача оборудования другому клиенту
    Given SN-55555 принадлежит «Банк Альфа»
    When Руководитель сервиса передаёт «Банк Бета» с датой 01.04.2026
    Then client_id = «Банк Бета»
    And в истории передач: от «Банк Альфа» к «Банк Бета», дата 01.04.2026
```

---

## Примечания

- Статус «На ремонте» устанавливается и снимается только автоматически (через заявки). Ручное изменение на этот статус недоступно.
- История передач хранится вечно (BR-R-009). Видна на вкладке «История передач» паспорта.
