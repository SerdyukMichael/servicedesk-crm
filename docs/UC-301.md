# UC-301: Управлять контактными лицами клиента

**Версия:** 1.0 | **Дата:** 31.03.2026 | **Статус:** Черновик

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-301 |
| **Модуль** | Модуль 3: CRM — Клиентская база |
| **Основной актор** | Менеджер по продажам |
| **Второстепенные акторы** | Руководитель сервиса, Администратор (те же права), Система (аудит-лог) |

| **Приоритет** | Must |
| **Предусловия** | Пользователь авторизован с ролью **Менеджер по продажам**, **Руководитель сервиса** или **Администратор**; карточка клиента-организации существует в системе |

## Основной поток — Добавление контактного лица

1. Пользователь открывает карточку клиента → вкладка **«Контактные лица»**.
2. Система отображает список существующих контактов (пустой при первом открытии).
3. Пользователь нажимает **«+ Добавить контакт»**.
4. Система отображает форму создания.
5. Пользователь заполняет обязательные поля: ФИО, роль/должность.
6. Опционально: телефон, email, является ли основным контактом (`is_primary`).
7. Пользователь нажимает **«Сохранить»**.
8. Система валидирует данные (уникальность email в рамках клиента, формат телефона).
9. Система создаёт запись с `is_active = true`, фиксирует в аудит-логе.
10. Система отображает **«Контакт добавлен»**.

---

## Основной поток — Редактирование контактного лица

1. Пользователь нажимает **«Редактировать»** у нужной строки.
2. Система отображает форму с заполненными значениями.
3. Пользователь вносит изменения → нажимает **«Сохранить»**.
4. Система валидирует, сохраняет, фиксирует в аудит-логе.

---

## Основной поток — Деактивация контактного лица

1. Пользователь нажимает **«Деактивировать»** у нужной строки.
2. Система показывает подтверждение: *«Деактивировать контакт [ФИО]? Он потеряет доступ к порталу (если был выдан)»*.
3. Пользователь подтверждает.
4. Система устанавливает `is_active = false`, фиксирует в аудит-логе.
5. Если у контакта была учётная запись портала — доступ блокируется (`portal_access = false`).

---

## Основной поток — Выдача / отзыв доступа к порталу самообслуживания

1. Пользователь нажимает **«Выдать доступ к порталу»** у нужного контакта.
2. Система открывает форму: email (подставляется из карточки, редактируемый), роль на портале (`portal_role`).
3. Пользователь подтверждает.
4. Система:
   - Ищет в таблице `users` запись с совпадающим email.
     - Если **не найдена** — создаёт новую запись с ролью `client_user`, `client_id = client_contacts.client_id`, временным паролем. Возвращает сгенерированный пароль в ответе API для передачи контакту.
     - Если **найдена и деактивирована** (`is_active = false`) — активирует (`is_active = true`), обновляет роль на `client_user` и `client_id`.
     - Если **найдена и активна** — обновляет роль и `client_id` при необходимости.
   - Устанавливает `portal_access = true` на контакте.
   - Сохраняет `portal_user_id` (FK → users.id) на контакте для обратной связи.
5. Система отображает **«Доступ выдан. Временный пароль: [пароль]»** (в MVP — до реализации email-приглашений).

> **Примечание MVP:** email-приглашение со ссылкой для установки пароля — будущий функционал (зависит от SMTP-интеграции). До его реализации временный пароль отображается единожды в интерфейсе.

**Отзыв доступа:**

1. Пользователь нажимает **«Отозвать доступ к порталу»**.
2. Система:
   - Устанавливает `portal_access = false` на контакте.
   - Деактивирует связанную учётную запись в `users` (`is_active = false`).
3. Контакт больше не может войти в систему.

---

## Альтернативные потоки

**АП-1: Смена основного контакта**
Только один контакт может быть `is_primary = true` в рамках клиента.
При установке флага у нового контакта система автоматически снимает его у предыдущего основного.

**АП-2: Просмотр без редактирования**
Директор и Сервисный инженер открывают вкладку «Контактные лица» в режиме только чтения. Кнопки «Добавить», «Редактировать», «Деактивировать» скрыты.

---

## Исключения

**ИП-1: Email уже используется другим контактом этого клиента**
→ Система блокирует: *«Email [адрес] уже назначен контакту [ФИО] этой организации»*.

**ИП-2: Попытка деактивировать основной контакт**
→ Система предупреждает: *«[ФИО] является основным контактом организации. После деактивации назначьте другой основной контакт»*. Деактивация не блокируется — только предупреждение.

**ИП-3: Попытка выдать доступ к порталу деактивированному контакту**
→ Система блокирует: *«Нельзя выдать доступ — контакт деактивирован. Сначала активируйте его»*.

**ИП-4: Email для портала не заполнен**
→ Кнопка «Выдать доступ к порталу» недоступна. Тултип: *«Для выдачи доступа укажите email контакта»*.

## Ключевые данные

### Таблица `client_contacts`

| Поле | Обязательное | Тип | Описание |
|---|---|---|---|
| id | Да | integer PK | — |
| client_id | Да | FK → clients.id, CASCADE | Организация-клиент |
| name | Да | varchar(128) | ФИО контактного лица |
| position | Нет | varchar(128) | Должность |
| phone | Нет | varchar(32) | Телефон |
| email | Нет | varchar(128) | Email (используется как логин на портале) |
| is_primary | Да | boolean, default false | Основной контакт организации |
| is_active | Да | boolean, default true | Активен ли контакт |
| portal_access | Да | boolean, default false | Есть ли доступ к порталу самообслуживания |
| portal_role | Нет | enum | `client_user` / `client_admin` — роль на портале |
| portal_user_id | Нет | FK → users.id, SET NULL | Связанная учётная запись в `users` (создаётся при выдаче доступа) |
| created_by | Да | FK → users.id | Кто создал |
| created_at | Да | timestamp | — |
| updated_at | Да | timestamp | — |

    Given «Петров П.П.» — основной контакт «Банк Альфа»

    Then система определяет её организацию «Банк Альфа» автоматически

  Scenario: Деактивация контакта блокирует доступ к порталу

    When Менеджер деактивирует контакт
    Then is_active=false

  Scenario: Блокировка дублирования email
    Given контакт с email «ivanova@alfa.ru» уже существует у «Банк Альфа»
    When Менеджер добавляет нового контакта с тем же email
    Then система показывает ошибку «Email уже назначен контакту Иванова А.С.»

  Scenario: Нельзя выдать доступ деактивированному контакту

## Примечания

- Контактные лица привязаны к организации-клиенту, а не к конкретной заявке. Один контакт может быть связан с несколькими заявками клиента.
- `portal_role = client_admin` зарезервировано для будущей версии, когда клиент-администратор сможет самостоятельно управлять другими контактами своей организации на портале.
- Деактивированные контакты скрыты из рабочих списков (`include_inactive=false` по умолчанию), но сохраняются в истории комментариев и заявок (BR-R-009).
- **Модуль 3 (CRM) вне MVP-scope реализации**, но данные для работы Модулей 9 (UC-901, UC-906) и 14 (уведомления клиентам) зависят от этой сущности. На этапе MVP создание контактов должно быть доступно как минимум через карточку клиента.

## Детали реализации (для разработчика)

### Миграция
`007_client_contacts_portal.py` — добавляет в `client_contacts`:
- `is_primary` boolean NOT NULL DEFAULT false
- `portal_access` boolean NOT NULL DEFAULT false
- `portal_role` ENUM('client_user', 'client_admin') NULL
- `created_by` FK → users.id SET NULL
- `created_at` / `updated_at` TIMESTAMP

### API эндпоинты (`/api/v1/clients/{client_id}/contacts`)

| Метод | URL | Роли | Описание |
|---|---|---|---|
| GET | `/contacts` | все авторизованные | Список контактов (`include_inactive=false`) |
| POST | `/contacts` | admin, sales_mgr, svc_mgr | Создать контакт |
| PUT | `/contacts/{id}` | admin, sales_mgr, svc_mgr | Редактировать |
| DELETE | `/contacts/{id}` | admin, sales_mgr | Деактивировать (soft-delete) |
| POST | `/contacts/{id}/portal-access` | admin, sales_mgr | Выдать доступ к порталу |
| DELETE | `/contacts/{id}/portal-access` | admin, sales_mgr | Отозвать доступ к порталу |

### Бизнес-правила в коде

- **Уникальность email**: при create и update проверяется среди активных контактов того же клиента → 409 CONFLICT.
- **is_primary**: при установке `is_primary=true` снимается у всех остальных контактов клиента (UPDATE WHERE client_id=X AND id!=new_id).
- **Деактивация**: `is_active=false` + `portal_access=false` (одновременно).
- **Портал**: если `portal_access=false` или `!is_active` — кнопка «Выдать доступ» недоступна на фронте; бэкенд возвращает 422 при попытке выдать деактивированному.

### Аудит-лог

Все операции пишут запись в `audit_log`:

| action | entity_type | когда |
|---|---|---|
| `create` | `client_contact` | создание контакта |
| `update` | `client_contact` | редактирование |
| `deactivate` | `client_contact` | деактивация |
| `portal_access_grant` | `client_contact` | выдача доступа к порталу |
| `portal_access_revoke` | `client_contact` | отзыв доступа |

Поле `created_by` на контакте фиксирует пользователя, создавшего запись (заполняется из JWT-токена).
