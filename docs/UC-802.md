# UC-802: Назначить / изменить роли и права доступа пользователя

**Версия:** 1.1 | **Дата:** 25.03.2026 | **Статус:** Согласован

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-802 |
| **Модуль** | Модуль 8: Учёт персонала |
| **Основной актор** | Администратор |
| **Второстепенные акторы** | Система (RBAC-движок, аудит-лог) |
| **Связанные BR-F** | BR-F-801, BR-F-806 |
| **Связанные BR-R** | BR-R-009 (soft delete), BR-R-010 (аудит-лог) |
| **Приоритет** | Must |
| **Предусловия** | Пользователь авторизован с ролью **Администратор**; карточка сотрудника существует (создана через UC-801); статус сотрудника — **Активен** |
| **Постусловия** | Сотруднику назначен набор ролей; его права доступа соответствуют объединению прав всех назначенных ролей (RBAC union); изменение зафиксировано в аудит-логе |
| **Триггер** | Администратор открывает карточку сотрудника и переходит в раздел «Роли и доступ» |

---

## Основной поток — Назначение ролей (первичное)

1. Администратор открывает карточку сотрудника → переходит на вкладку **«Роли и доступ»**.
2. Система отображает текущее состояние: роли не назначены, доступ в систему закрыт.
3. Администратор нажимает **«Назначить роли»**.
4. Система отображает список ролей с чекбоксами: Администратор, Менеджер по продажам, Руководитель сервиса, Сервисный инженер, Кладовщик, Бухгалтер, Директор.
5. Администратор выбирает одну или несколько ролей.
6. Система отображает **сводный preview прав**: объединение прав всех выбранных ролей с подсветкой конфликтов (если есть).
7. Администратор нажимает **«Применить»**.
8. Система сохраняет набор ролей, генерирует учётные данные / отправляет приглашение на email.
9. Система фиксирует в аудит-логе (`action: roles_assign`, `user_id`, `roles: [список]`).
10. Система отображает **«Роли назначены. Приглашение отправлено на [email]»**.

---

## Основной поток — Изменение набора ролей

1. Администратор открывает карточку → вкладка **«Роли и доступ»**.
2. Система отображает текущий набор ролей (чекбоксы с отмеченными активными).
3. Администратор добавляет или снимает нужные роли.
4. Система обновляет preview сводных прав в реальном времени.
5. Администратор нажимает **«Применить»**.
6. Система обновляет набор ролей, немедленно перестраивает права (активные сессии получают новые права при следующем запросе).
7. Система фиксирует в аудит-логе (`action: roles_update`, `added_roles: [...]`, `removed_roles: [...]`).
8. Система отображает **«Роли обновлены»**.

---

## Альтернативные потоки

**АП-1: Отзыв доступа без снятия ролей**
Администратор нажимает **«Отозвать доступ»**.
Система устанавливает `is_active = false`, завершает все активные сессии.
Роли сохраняются — при восстановлении доступа набор ролей не нужно назначать заново.
Аудит-лог: `action: access_revoked`.

**АП-2: Восстановление доступа**
Администратор нажимает **«Восстановить доступ»**.
Система устанавливает `is_active = true`, отправляет ссылку сброса пароля.
Прежний набор ролей восстанавливается автоматически.
Аудит-лог: `action: access_restored`.

---

## Исключения

**ИП-1: Снятие всех ролей**
Администратор пытается сохранить изменения, не выбрав ни одной роли.
→ Система блокирует: *«Необходимо выбрать хотя бы одну роль. Чтобы заблокировать доступ — используйте "Отозвать доступ"»*.

**ИП-2: Удаление роли Администратора — последний в системе**
При попытке снять роль «Администратор» у единственного активного администратора:
→ Система блокирует: *«В системе должен быть хотя бы один пользователь с ролью Администратор»*.

**ИП-3: Попытка изменить роли уволенного сотрудника**
→ Система блокирует: *«Невозможно изменить роли — сотрудник имеет статус "Уволен"»*.

**ИП-4: Сотрудник авторизован в момент смены ролей**
Новые права применяются к следующему API-запросу. При переходе на ставшую недоступной страницу — ошибка 403: *«Ваши права доступа изменились. Обновите страницу»*.

## Ключевые данные

| Поле | Тип | Описание |
|---|---|---|
| roles | `[]enum` | Массив назначенных ролей (1..N). Права = объединение прав всех ролей |
| is_active | boolean | Активна ли учётная запись (независимо от ролей) |
| invited_at | timestamp | Дата отправки приглашения |
| last_login_at | timestamp | Дата последнего входа (read-only) |

> Итоговые права пользователя = **union** всех ролей. Если роль А разрешает действие X, а роль Б — нет, действие X **разрешено** (разрешение приоритетнее запрета).

## Acceptance Criteria

```gherkin
Feature: Назначение и управление ролями пользователей

  Scenario: Назначение нескольких ролей одновременно
    Given сотрудник существует со статусом «Активен» без ролей
    When Администратор выбирает роли «Сервисный инженер» и «Кладовщик» и нажимает «Применить»
    Then сотруднику назначены обе роли
    And его права = объединение прав обеих ролей
    And в аудит-логе: action=roles_assign, roles=[Сервисный инженер, Кладовщик]

  Scenario: Добавление роли к существующему набору
    Given сотрудник имеет роль «Менеджер по продажам»
    When Администратор добавляет роль «Бухгалтер» и нажимает «Применить»
    Then сотрудник имеет роли [Менеджер по продажам, Бухгалтер]
    And в аудит-логе: action=roles_update, added_roles=[Бухгалтер], removed_roles=[]

  Scenario: Снятие одной роли из набора
    Given сотрудник имеет роли [Сервисный инженер, Кладовщик]
    When Администратор снимает роль «Кладовщик» и нажимает «Применить»
    Then сотрудник имеет только роль «Сервисный инженер»
    And в аудит-логе: action=roles_update, added_roles=[], removed_roles=[Кладовщик]

  Scenario: Блокировка сохранения без ролей
    Given сотрудник имеет роль «Кладовщик»
    When Администратор снимает все роли и нажимает «Применить»
    Then система показывает ошибку «Необходимо выбрать хотя бы одну роль»
    And изменения не сохраняются

  Scenario: Блокировка удаления последнего Администратора
    Given только один активный пользователь с ролью «Администратор»
    When Администратор снимает с себя роль «Администратор» и нажимает «Применить»
    Then система показывает ошибку «В системе должен быть хотя бы один пользователь с ролью Администратор»

  Scenario: Отзыв и восстановление доступа с сохранением ролей
    Given сотрудник имеет роли [Сервисный инженер, Кладовщик] и активный доступ
    When Администратор нажимает «Отозвать доступ»
    Then is_active = false, сессии завершены, роли сохранены
    When Администратор нажимает «Восстановить доступ»
    Then is_active = true, роли [Сервисный инженер, Кладовщик] восстановлены автоматически
```

## Примечания / Бизнес-правила

- **BR-F-801:** поддерживается 7 фиксированных ролей. Один сотрудник может иметь **любое количество ролей одновременно** (совмещение).
- **BR-F-806:** итоговые права = **union** прав всех назначенных ролей. Жёстко зашитые ограничения (BR-R-001 и др.) не снимаются даже при совмещении ролей.
- **BR-R-009:** роли уволенного сотрудника не удаляются физически — хранятся в аудит-логе.
- Разница между **«нет ролей»** и **«отозван доступ»**: нет ролей — технически невозможное состояние (блокируется системой); отозван доступ — роли есть, `is_active = false`.
- Связь с UC-801: роль назначается только после создания карточки.
