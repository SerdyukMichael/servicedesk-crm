# UC-805: Управлять графиком отпусков и отсутствий сотрудника

**Версия:** 2.0 | **Дата:** 27.03.2026 | **Статус:** На проверке

> **v2.0:** добавлен поток редактирования периода, поле `source` для будущей синхронизации с 1С, доступ Директора (просмотр), ИП для уволенного сотрудника.

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-805 |
| **Модуль** | Модуль 8: Учёт персонала |
| **Основной актор** | Руководитель сервиса |
| **Второстепенные акторы** | Администратор (те же права), Директор (только просмотр), Система (предупреждение при назначении заявки) |
| **Связанные BR-F** | BR-F-808, BR-F-810 |
| **Связанные BR-R** | BR-R-010 (аудит-лог) |

| **Постусловия** | Период отсутствия зафиксирован; при назначении заявки на этого сотрудника система будет показывать предупреждение, если дата заявки попадает в период отсутствия |
| **Триггер** | Пользователь открывает карточку сотрудника и переходит на вкладку **«Отпуска и отсутствия»** |

## Основной поток — Добавление периода отсутствия

1. Пользователь открывает карточку сотрудника → вкладка **«Отпуска и отсутствия»**.
2. Система отображает список существующих периодов (пустой или с ранее добавленными).
3. Пользователь нажимает **«+ Добавить период»**.
4. Система отображает форму с полями: тип отсутствия, дата начала, дата окончания, комментарий (опционально).
5. Пользователь заполняет форму и нажимает **«Сохранить»**.
6. Система валидирует данные (дата начала ≤ дата окончания; нет пересечения с существующими периодами).
7. Система сохраняет запись с `source = manual`, фиксирует в аудит-логе (`action: create`, `entity: absence`).
8. Система обновляет список и устанавливает статус сотрудника **«В отпуске»**, если период включает текущую дату.

---

## Основной поток — Редактирование периода отсутствия

1. Пользователь нажимает **«Редактировать»** у нужной строки в списке.
2. Система открывает форму с заполненными текущими значениями.
   - Записи с `source = 1c_sync` открываются только для чтения (см. АП-2).
3. Пользователь изменяет нужные поля и нажимает **«Сохранить»**.
4. Система валидирует данные (те же правила, что при добавлении; при проверке пересечений текущая запись исключается из сравнения).
5. Система сохраняет изменения, пересчитывает статус сотрудника, фиксирует в аудит-логе (`action: update`, старое и новое значение).
6. Система показывает сообщение **«Изменения сохранены»**.

---

## Основной поток — Удаление периода отсутствия

1. Пользователь нажимает **«Удалить»** у нужной строки.
2. Система показывает подтверждение: *«Удалить период отсутствия [дд.мм.гггг – дд.мм.гггг]?»*.
3. Пользователь подтверждает.
4. Система удаляет запись (soft delete), фиксирует в аудит-логе (`action: delete`, `entity: absence`).
5. Если удалён текущий активный период — статус сотрудника возвращается на **«Активен»**.

---

## Альтернативные потоки

**АП-1: Предупреждение при назначении заявки на сотрудника в отпуске**
При попытке назначить заявку на инженера, дата выезда которой попадает в период отсутствия:
→ Система показывает предупреждение: *«Инженер [ФИО] в отпуске с [дата] по [дата]. Назначить всё равно?»*
→ Назначение не блокируется — только информирование. Пользователь подтверждает или выбирает другого инженера.

**АП-2: Записи с source = 1c_sync (будущая версия)**
В текущей версии все записи создаются вручную (`source = manual`). Поле `source` зарезервировано для будущей интеграции с 1С.
Когда синхронизация будет реализована: записи с `source = 1c_sync` будут отображаться в режиме только чтения — кнопки «Редактировать» и «Удалить» для них заблокированы. Тултип: *«Период синхронизирован из 1С. Для изменения скорректируйте данные в 1С»*.

**АП-3: Просмотр без редактирования (Директор)**
Директор может открыть вкладку «Отпуска и отсутствия» в режиме только чтения. Кнопки «+ Добавить», «Редактировать», «Удалить» скрыты.

---

## Исключения

**ИП-1: Пересечение периодов**
На шаге 6: новый или изменённый период пересекается с уже существующим.
→ Система блокирует: *«Пересечение с существующим периодом [дд.мм.гггг – дд.мм.гггг]. Скорректируйте даты»*.

**ИП-2: Дата начала позже даты окончания**
→ Система блокирует: *«Дата начала должна быть раньше даты окончания»*.

**ИП-3: Сотрудник не активен**
Пользователь открывает вкладку «Отпуска и отсутствия» у сотрудника со статусом **Уволен**.
→ Вкладка отображается в режиме только чтения. Кнопки добавления и редактирования скрыты. Плашка: *«Редактирование недоступно: сотрудник уволен»* (BR-F-810).

## Примечания / Бизнес-правила

- **BR-F-808:** предупреждение при назначении заявки на инженера с активным периодом отсутствия — информационное, не блокирующее.
- **BR-F-810:** добавление и редактирование периодов доступно только для сотрудников со статусом `Активен`. Для уволенных (`is_deleted = true`) — только просмотр.
- Статус сотрудника «В отпуске» в карточке (UC-801) и периоды в этом UC — независимые механизмы. Статус можно выставить вручную; период — плановый инструмент для заблаговременного внесения графика.
- Прошедшие периоды отображаются в архивном разделе вкладки (не удаляются).
- **Синхронизация с 1С** (планируется): поле `source` зарезервировано. Записи с `source = 1c_sync` управляются исключительно через интеграцию.
