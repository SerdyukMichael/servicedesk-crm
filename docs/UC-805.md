## Основной поток — Добавление периода отсутствия

1. Руководитель сервиса открывает карточку сотрудника → вкладка **«Отпуска и отсутствия»**.
2. Система отображает список существующих периодов (пустой или с ранее добавленными).
3. Руководитель сервиса нажимает **«+ Добавить период»**.
4. Система отображает форму с полями: тип отсутствия, дата начала, дата окончания, комментарий (опционально).
5. Руководитель сервиса заполняет форму и нажимает **«Сохранить»**.
6. Система валидирует данные (дата начала ≤ дата окончания; нет пересечения с существующими периодами).
7. Система сохраняет запись, фиксирует в аудит-логе.
8. Система обновляет список и устанавливает статус сотрудника **«В отпуске»** если период включает текущую дату.

---

## Основной поток — Удаление периода отсутствия

1. Руководитель сервиса нажимает **«Удалить»** у нужной строки.
2. Система показывает подтверждение: *«Удалить период отсутствия [даты]?»*.
3. Руководитель сервиса подтверждает.
4. Система удаляет запись (soft delete), фиксирует в аудит-логе.
5. Если удалён текущий активный период — статус сотрудника возвращается на **«Активен»**.

---

## Альтернативные потоки

**АП-1: Предупреждение при назначении заявки на сотрудника в отпуске**
При попытке назначить заявку на инженера, дата выезда которой попадает в период отсутствия:
→ Система показывает предупреждение: *«Инженер [ФИО] в отпуске с [дата] по [дата]. Назначить всё равно?»*
→ Назначение не блокируется — только информирование. Руководитель сервиса подтверждает или выбирает другого инженера.

---

## Исключения

**ИП-1: Пересечение периодов**
На шаге 6: новый период пересекается с уже существующим.
→ Система блокирует: *«Пересечение с существующим периодом [даты]. Скорректируйте даты»*.

**ИП-2: Дата начала позже даты окончания**
→ Система блокирует: *«Дата начала должна быть раньше даты окончания»*.

## Acceptance Criteria

```gherkin
Feature: Управление графиком отпусков и отсутствий

  Scenario: Успешное добавление периода отпуска
    Given Руководитель сервиса открыл вкладку «Отпуска и отсутствия» инженера «Иванов»
    When он добавляет период «Отпуск» с 01.04.2026 по 14.04.2026 и нажимает «Сохранить»
    Then период появляется в списке
    And если сегодняшняя дата попадает в этот период — статус инженера меняется на «В отпуске»

  Scenario: Предупреждение при назначении заявки в период отпуска
    Given инженер «Иванов» имеет период отпуска с 01.04.2026 по 14.04.2026
    When Руководитель сервиса назначает заявку с датой выезда 05.04.2026 на «Иванова»
    Then система показывает предупреждение «Инженер Иванов в отпуске с 01.04.2026 по 14.04.2026»
    And назначение не блокируется — доступна кнопка «Назначить всё равно»

  Scenario: Блокировка при пересечении периодов
    Given инженер «Иванов» имеет период с 01.04.2026 по 14.04.2026
    When добавляется новый период с 10.04.2026 по 20.04.2026
    Then система показывает ошибку «Пересечение с существующим периодом»
    And сохранение блокируется
```

## Примечания / Бизнес-правила

- **BR-F-808:** предупреждение при назначении — информационное, не блокирующее.
- Статус сотрудника «В отпуске» в карточке (UC-801) и периоды в этом UC — независимые механизмы. Статус можно поставить вручную; период — плановый инструмент для заблаговременного внесения графика.
- Прошедшие периоды отображаются в архивном разделе вкладки (не удаляются).
