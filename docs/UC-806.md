# UC-806: Просмотреть аудит-лог действий пользователей

**Версия:** 1.0 | **Дата:** 26.03.2026 | **Статус:** Рабочий

---

## Паспорт

| Поле | Значение |
|---|---|
| **ID** | UC-806 |
| **Модуль** | Модуль 8: Учёт персонала |
| **Основной актор** | Администратор |
| **Второстепенные акторы** | Директор (только просмотр) |
| **Связанные BR-F** | BR-F-807 |
| **Связанные BR-R** | BR-NF-014 (хранение ≥ 2 лет, логи неизменяемы) |
| **Приоритет** | Must |
| **Предусловия** | Пользователь авторизован с ролью **Администратор** или **Директор** |
| **Постусловия** | Нет изменений в данных — только чтение. Экспорт опционален |
| **Триггер** | Администратор открывает раздел **«Аудит-лог»** |

---

## Основной поток

1. Администратор переходит в раздел **«Аудит-лог»**.
2. Система отображает таблицу событий с сортировкой по дате (новые сверху).
3. Администратор применяет фильтры при необходимости (см. АП-1).
4. Администратор просматривает события.
5. При необходимости — раскрывает детали события (кнопка «Подробнее»).
6. При необходимости — экспортирует результаты (см. АП-2).

---

## Состав записи аудит-лога

| Поле | Описание |
|---|---|
| Дата / Время | Timestamp в UTC с точностью до секунды |
| Пользователь | ФИО + роль пользователя, выполнившего действие |
| Тип действия | create / update / delete / login / logout / roles_assign / roles_update / access_revoked и др. |
| Объект | Сущность (user, ticket, equipment, invoice, ...) |
| ID объекта | Идентификатор записи |
| Описание | Краткое описание: что изменилось (поле, старое → новое значение) |
| IP-адрес | IP источника запроса |

---

## Альтернативные потоки

**АП-1: Фильтрация лога**
Доступные фильтры:
- По пользователю (выбор из списка)
- По типу действия (мультивыбор)
- По модулю / объекту (user, ticket, equipment, invoice, ...)
- По диапазону дат
- По IP-адресу (текстовый поиск)

Все фильтры применяются одновременно (AND-логика).

**АП-2: Экспорт в CSV**
Администратор нажимает **«Экспорт»**.
Система формирует CSV-файл с текущей выборкой (с учётом активных фильтров) и предлагает скачать.
Экспорт не требует дополнительной авторизации.

---

## Исключения

**ИП-1: Попытка изменить или удалить запись лога**
→ Интерфейс не предоставляет такой возможности. На уровне БД записи аудит-лога — append-only (BR-NF-014).

**ИП-2: Запрос слишком большого периода (> 1 года)**
→ Система предупреждает: *«Запрос охватывает большой период. Это может занять некоторое время»* и выполняет запрос с пагинацией (100 записей на страницу).

## Acceptance Criteria

```gherkin
Feature: Аудит-лог действий пользователей

  Scenario: Событие создания сотрудника фиксируется в логе
    Given Администратор создал нового сотрудника «Петров»
    When Администратор открывает аудит-лог с фильтром entity=user, action=create
    Then в списке есть запись: actor=[Администратор], entity=user, entity_id=[ID Петрова], action=create

  Scenario: Фильтрация по пользователю
    Given в системе 100 записей аудит-лога от разных пользователей
    When Администратор фильтрует по пользователю «Иванов»
    Then отображаются только записи, где actor = «Иванов»

  Scenario: Экспорт выборки в CSV
    Given Администратор применил фильтр: entity=ticket, диапазон дат = март 2026
    When нажимает «Экспорт»
    Then скачивается CSV-файл, содержащий только записи, соответствующие фильтру

  Scenario: Записи лога нельзя изменить или удалить
    Given в аудит-логе есть запись action=create для ticket ID=42
    Then в интерфейсе отсутствуют кнопки «Изменить» и «Удалить» для любой записи лога

  Scenario: Детализация записи
    Given в логе есть запись action=update, entity=user
    When Администратор нажимает «Подробнее» на этой записи
    Then система показывает: поле, которое было изменено, старое значение и новое значение
```

## Примечания / Бизнес-правила

- **BR-F-807:** аудит-лог ведётся автоматически для всех модулей системы. Пользователь не может отключить логирование.
- **BR-NF-014:** хранение логов — не менее 2 лет. Записи неизменяемы (append-only). Удаление через интерфейс или API запрещено.
- Аудит-лог доступен только Администратору и Директору. Другие роли не имеют доступа к разделу.
- Системные события (фоновые задачи, авто-эскалации, авто-создание ТО) фиксируются с actor_id = NULL и actor_type = 'system'.
