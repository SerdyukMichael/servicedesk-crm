# Release v0.9.0 — Подписание акта, статус оплаты, защита редактирования

**Дата:** 2026-04-09

## Что изменилось

Реализованы бизнес-правила BR-F-115 – BR-F-120:
- Подписание акта — только представитель клиента (`client_user`)
- Кнопка «Редактировать акт» скрывается после подписания
- Кнопка «Создать счёт» блокируется если в акте нет позиций
- Рядом с актом отображается ссылка на связанный счёт и статус оплаты
- Страница счетов: имя клиента и колонка «Оплачен»

---

## Backend

### Новое
- `POST /api/v1/tickets/{ticket_id}/work-act/sign` — упрощённый эндпоинт подписания без явного `act_id`; доступен только роли `client_user`
- Фильтр `ticket_id` в `GET /api/v1/invoices` — выборка счетов по заявке
- Поле `is_paid: bool` в `InvoiceResponse` — вычисляется из `status == "paid"`

### Изменено
- `POST /api/v1/tickets/{ticket_id}/work-act/{act_id}/sign` — ограничен ролью `client_user` (ранее: `svc_mgr`, `admin`, `client_user`)

### Тесты
- `test_comments_and_sign.py`: тест `test_svc_mgr_can_still_sign_act` → `test_svc_mgr_cannot_sign_act` (проверяет 403)
- Итого: **364 теста, все зелёные**

---

## Frontend

### TicketDetailPage
- `canSignAct` — только `hasRole('client_user')`
- Статус «Подпись клиента»: «✓ Подписан [дата]» (зелёный) или «Не подписан»
- Строка «Счёт» рядом с актом: ссылка + badge статуса (draft / ✓ Оплачен)
- Кнопка «Редактировать акт» скрыта если `workAct.signed_by` заполнен
- Кнопка «Создать счёт из акта» `disabled` если `items.length === 0`; скрыта если счёт уже создан
- Запрос счетов по `ticket_id` через `useQuery(['invoices','by-ticket',ticketId])`
- Инвалидация кэша счетов после создания

### InvoicesPage
- Имя клиента: маппинг `client_id → clients[]` (исправлен баг, когда показывалось «—»)
- Новая колонка «ОПЛАЧЕН»: «✓ Да» (зелёный) или «—»

### API
- `getInvoicesByTicket(ticketId)` — новый хелпер
- `Invoice.is_paid: boolean` — новое поле в типе
